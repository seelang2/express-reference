Introduction

This is a production-oriented reference guide for building web applications with Node.js and Express. It was developed through a series of in-depth explorations covering the full application stack — each section goes beyond syntax to explain the reasoning behind decisions, the tradeoffs between approaches, and the failure modes to watch for in production. Code examples reflect current best practices as of 2025.

The guide is intended to be consulted during development rather than read front to back. If you're setting up a new project, start with Setup. If you're hardening an existing app, jump to Security or Supply chain. If something is failing in production and you're not sure why, Observability is where you want to be.

Conventions used throughout

All examples use TypeScript with strict: true, ESM (type: "module"), and Node.js v22 LTS. Package choices reflect current community consensus — jose over jsonwebtoken, csrf-csrf over the deprecated csurf, argon2id over bcrypt, shx for cross-platform build scripts. Where a package has been deprecated or has known vulnerabilities, that is called out explicitly.

This guide is a living document. Sections will be added and updated as the stack evolves and new topics are covered.

Project initialization

Start every project with ESM and modern Node. Avoid CommonJS for new projects — import/export is the standard and CommonJS interop gets messier over time, not easier.

Bootstrap

mkdir my-api && cd my-api
node --version      # aim for LTS (v22+)
npm init -y
npm pkg set type="module"

Core dependencies

npm i express
npm i -D nodemon tsx typescript @types/node @types/express shx

tsx runs .ts files directly without a build step in dev. shx wraps Unix shell commands so your npm scripts work identically on Mac, Windows, and Linux CI runners without PowerShell workarounds.

package.json scripts

{
  "type": "module",
  "scripts": {
    "build":        "npm run clean && tsc && npm run copy:static",
    "copy:static":  "shx cp -r src/static dist/static",
    "clean":        "shx rm -rf dist",
    "dev":          "nodemon --exec tsx src/index.ts",
    "start":        "node dist/index.js"
  }
}
Use TypeScript by default. The upfront cost is tiny; the long-term gain in refactoring safety, autocomplete, and caught errors is large. strict: true in tsconfig catches an entire class of bugs at compile time.

tsconfig essentials

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true
  }
}

module: "Node16" and moduleResolution: "Node16" work together to enforce that all local imports include file extensions (./users.service.js) — which is required for ESM in Node. This trips people up initially but reflects how Node actually resolves modules at runtime.

Project structure

Organize by feature, not by file type. A flat controllers/ + models/ layout feels organized at first but breaks down once you have more than a handful of features — every change touches multiple directories, and there's no clear ownership of a feature's behaviour.

Feature folders keep cohesion high: everything related to orders lives in features/orders/. Coupling is explicit: if orders imports from users, you can see it clearly.

The two-file server pattern

app.ts configures Express (registers middleware, routes, error handler) but never calls listen(). index.ts imports the app and starts the server. This separation means test files can import app directly and make HTTP assertions without binding a real port — a small decision that makes every integration test cleaner.

Config & environment

Never scatter process.env.X calls throughout your codebase. Validate everything at startup in one place and export a typed config object. If any variable is missing or malformed, the process exits immediately with a clear error — not silently at runtime when a request hits a broken code path an hour after deployment.

// src/config/index.ts
import { z } from 'zod'

const schema = z.object({
  NODE_ENV:            z.enum(['development', 'test', 'production']),
  PORT:                z.coerce.number().default(3000),
  DB_URL:              z.string().url(),
  JWT_ACCESS_SECRET:   z.string().min(32),
  JWT_REFRESH_SECRET:  z.string().min(32),
  CSRF_SECRET:         z.string().min(32),
  COOKIE_SECRET:       z.string().min(32),
  ALLOWED_ORIGINS:     z.string(),
})

export const config = schema.parse(process.env)

if (config.NODE_ENV === 'production' && !config.ALLOWED_ORIGINS) {
  throw new Error('ALLOWED_ORIGINS must be explicitly set in production')
}

Static files

tsc only processes .ts files. It ignores everything else — HTML, CSS, images, fonts. You need a parallel copy step that runs alongside the TypeScript compiler so static assets land in dist/ in the right location.

src/static/ is the committed source of truth. dist/ is entirely generated and git-ignored. The build system copies src/static/ to dist/static/ so Express can serve it at runtime.

Express serving

import { fileURLToPath } from 'url'
import { join, dirname } from 'path'

// ESM doesn't provide __dirname natively — this pattern is required

app.use(express.static(join(__dirname, 'static', 'public'), {
  maxAge:  '1d',
  etag:    true,
  index:   'index.html',
}))

Server setup

Keep app.ts clean and in a deliberate order: security middleware first, then body parsing, then logging, then routes, then the catch-all 404, then the error handler last. The error handler must be the final app.use — Express identifies it by its four-argument signature.

app.ts

import express from 'express'
import helmet from 'helmet'
import cors from 'cors'
import cookieParser from 'cookie-parser'
import { corsOptions } from './config/cors.js'
import { ordersRouter } from './features/orders/index.js'
import { errorHandler } from './middleware/errorHandler.js'
import { requestLogger } from './middleware/requestLogger.js'
import { config } from './config/index.js'

export const app = express()

// Exact number of proxy hops between internet and this process
// Required for req.ip and rate limiting to work correctly behind Nginx
app.set('trust proxy', 1)

app.use(helmet({ /* see Security section */ }))
app.use(cors(corsOptions))
app.options('*', cors(corsOptions))         // handle preflight for all routes
app.use(cookieParser(config.COOKIE_SECRET))
app.use(express.json({ limit: '100kb' }))
app.use(express.urlencoded({ extended: true, limit: '100kb' }))
app.use(requestLogger)

app.use('/api/v1/orders', ordersRouter)
// ... other feature routers

app.use((req, res) => res.status(404).json({ code: 'NOT_FOUND' }))
app.use(errorHandler)

index.ts

import { createServer } from 'http'
import { app } from './app.js'
import { db } from './lib/db.js'
import { config } from './config/index.js'

const server = createServer(app)

// Prevent slow-loris and connection exhaustion attacks
server.headersTimeout   = 20_000
server.requestTimeout   = 30_000
server.keepAliveTimeout = 5_000

async function start() {
  if (config.NODE_ENV === 'production') {
    const { execSync } = await import('child_process')
    execSync('npx prisma migrate deploy', { stdio: 'inherit' })
  }
  await db.$connect()
  // Bind to 127.0.0.1 — only reachable via Nginx, never directly
  server.listen(config.PORT, '127.0.0.1', () => {
    console.log(`Ready on port ${config.PORT}`)
  })
}

async function shutdown() {
  server.close()
  await db.$disconnect()
  process.exit(0)
}

process.on('SIGTERM', shutdown)
process.on('SIGINT',  shutdown)
process.on('unhandledRejection', (err) => {
  console.error('Unhandled rejection:', err)
  process.exit(1)
})

start()
Always handle SIGTERM. Kubernetes and Docker send it before killing your container. server.close() stops accepting new connections and waits for in-flight requests to finish before the process exits.

Routing

Routers are purely declarative — they express which method, which path, which middleware runs first, and which controller handles the request. No logic lives in a router file.

Router

// features/orders/orders.router.ts
import { Router } from 'express'
import { authenticate, authorize } from '../../middleware/auth.js'
import * as ctrl from './orders.controller.js'

export const ordersRouter = Router()

ordersRouter.get('/',        authenticate, ctrl.listOrders)
ordersRouter.get('/:id',     authenticate, ctrl.getOrder)
ordersRouter.post('/',       authenticate, ctrl.createOrder)
ordersRouter.patch('/:id',   authenticate, ctrl.updateOrder)
ordersRouter.delete('/:id',  authenticate, authorize('admin'), ctrl.deleteOrder)

Controller — the HTTP translator

Each handler does exactly three things: extract from req, delegate to the service with plain data, respond with the result. HTTP decisions (status codes, response shape) live here. Domain decisions do not.

// orders.controller.ts
export async function createOrder(req, res, next) {
  try {
    const body   = createOrderSchema.parse(req.body)
    const userId = req.user.id
    const order  = await ordersService.createOrder(userId, body)
    res.status(201).json(order)
  } catch (err) { next(err) }
}

export async function cancelOrder(req, res, next) {
  try {
    await ordersService.cancelOrder(req.params.id, req.user.id)
    res.status(204).send()
  } catch (err) { next(err) }
}
Each handler function is what stays short (~10 lines), not the entire controller file. A controller with 5 CRUD handlers is naturally 50–70 lines. The moment you write an if about a domain rule inside a controller, that logic belongs in the service.
DON'T — logic in a controller

Checking inventory, calculating totals, enforcing business rules, querying the DB — all of this belongs in the service layer.

Middleware

Middleware is a function with the signature (req, res, next). Write it as a named, exported function — not an inline anonymous arrow. Named functions appear in stack traces, can be unit tested independently, and communicate intent clearly.

Request logger (structured)

export function requestLogger(req, res, next) {
  const start = Date.now()
  res.on('finish', () => {
    logger.info({
      method:    req.method,
      path:      req.path,
      status:    res.statusCode,
      ms:        Date.now() - start,
      requestId: req.id,
    })
  })
  next()
}

Request ID — for distributed tracing

import { randomUUID } from 'crypto'

app.use((req, res, next) => {
  req.id = (req.headers['x-request-id'] as string) ?? randomUUID()
  res.setHeader('x-request-id', req.id)
  next()
})

Pass req.id into every log call and DB query. This lets you trace a full request through your logs — invaluable when debugging production issues.

Async patterns

Express 4 does not catch async errors automatically. An unhandled promise rejection inside a route handler crashes the process in older Node versions or silently swallows the error. You must either use a wrapper or upgrade to Express 5.

Express 4 — asyncHandler wrapper

// lib/asyncHandler.ts
import type { RequestHandler } from 'express'

export function asyncHandler(fn: RequestHandler): RequestHandler {
  return (req, res, next) =>
    Promise.resolve(fn(req, res, next)).catch(next)
}

// Usage — errors automatically forwarded to error handler
export const getUser = asyncHandler(async (req, res) => {
  const user = await usersService.getById(req.params.id)
  if (!user) throw new NotFoundError()
  res.json(user)
})

Express 5 — recommended for new projects

npm i express@5

// Async errors are automatically forwarded — no wrapper needed
app.get('/users/:id', async (req, res) => {
  const user = await usersService.getById(req.params.id)
  if (!user) throw new NotFoundError()
  res.json(user)
})
Even with Express 5, keep your global error handler and handle process.on('unhandledRejection'). Errors outside the request lifecycle (in background workers, DB connection handlers, scheduled tasks) still need explicit handling.

Error handling

Define a custom error class hierarchy. Your global error handler maps these to HTTP responses. Everything else maps to 500. The key principle: errors flow upward via next(err) from anywhere in the stack and exit through one single handler — never scattered res.status(500).send() calls throughout controllers.

Custom error classes

// lib/errors.ts
export class AppError extends Error {
  constructor(
    public statusCode: number,
    public code: string,
    message: string
  ) { super(message) }
}

export class NotFoundError     extends AppError { constructor(m = 'Not found')     { super(404, 'NOT_FOUND', m) } }
export class UnauthorizedError extends AppError { constructor(m = 'Unauthorized')  { super(401, 'UNAUTHORIZED', m) } }
export class ForbiddenError    extends AppError { constructor(m = 'Forbidden')     { super(403, 'FORBIDDEN', m) } }
export class ConflictError     extends AppError { constructor(m = 'Conflict')      { super(409, 'CONFLICT', m) } }
export class ValidationError   extends AppError { constructor(m = 'Invalid input') { super(422, 'VALIDATION_ERROR', m) } }

Global error handler — must be last middleware in app.ts

import type { ErrorRequestHandler } from 'express'
import { ZodError } from 'zod'

export const errorHandler: ErrorRequestHandler = (err, req, res, _next) => {
  if (err instanceof ZodError) {
    return res.status(422).json({
      code:   'VALIDATION_ERROR',
      errors: err.flatten()
    })
  }
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({
      code:    err.code,
      message: err.message
    })
  }
  logger.error({ err, requestId: req.id }, 'Unhandled error')
  res.status(500).json({ code: 'INTERNAL_ERROR' })
}
Never send err.stack, internal error messages, or DB error details to clients in production. An attacker can use stack traces and SQL error messages to understand your schema and infrastructure.

Router → Controller → Service

This is the pattern that underpins the entire module design. Each layer has a precisely defined job and knows nothing about the layers above it.

The three rules of a controller method

1 — Extract

Pull data from req: params, validated body, query string, req.user from auth middleware. Nothing else.

2 — Delegate

Call the service with plain data. Never pass req or res into a service — this would couple your business logic to Express.

3 — Respond

Set the HTTP status code and serialize the result. Status code decisions (201 vs 200, 204 no body) are HTTP concerns and live here.

A complete feature — POST /orders

// orders.router.ts — declarative only
ordersRouter.post('/', authenticate, ctrl.createOrder)

// orders.controller.ts — HTTP translation, ~8 lines
export async function createOrder(req, res, next) {
  try {
    const body  = createOrderSchema.parse(req.body)
    const order = await ordersService.createOrder(req.user.id, body)
    res.status(201).json(order)
  } catch (err) { next(err) }
}

// orders.service.ts — all the real decisions
export async function createOrder(userId, input) {
  for (const item of input.items) {
    const product = await productsRepo.findById(item.productId)
    if (!product)                      throw new NotFoundError(`Product not found`)
    if (product.stock < item.quantity) throw new ConflictError('Insufficient stock')
  }
  const total = input.items.reduce((s, i) => s + i.quantity * i.unitPrice, 0)
  return db.$transaction(async (tx) => {
    const order = await ordersRepo.createWithItems(userId, input, total, tx)
    for (const item of input.items)
      await productsRepo.decrementStock(item.productId, item.quantity, tx)
    return order
  })
}

The litmus test: could you call ordersService.createOrder() from a CLI script, a cron job, or a WebSocket handler? If yes — no req passed in, no Express imports — the separation is clean.

Barrel exports

The barrel (index.ts) is the public API for consumers outside the feature. Think of it as the feature's package boundary — what it publishes vs what it keeps private.

Internal imports — direct, always fine

// orders.controller.ts — inside the feature, imports siblings directly
import { createOrderSchema } from './orders.schema.js'
import * as ordersService from './orders.service.js'

External imports — barrel only

// app.ts — outside the feature
import { ordersRouter } from './features/orders/index.js'

// Another feature needing the Order type
import type { Order } from './features/orders/index.js'

The barrel — selective, editorial

// features/orders/index.ts
export { ordersRouter } from './orders.router.js'
export type { Order, CreateOrderInput } from './orders.schema.js'

// NOT exported: service, controller, repository, schema internals

Classes vs functions

The current best-practice lean in the Express community is plain functions for controllers and class-based services when testability matters.

Controllers — why functions win in plain Express

// PROBLEM with class-based controllers
class OrdersController {
  async createOrder(req, res, next) { ... }
}
const controller = new OrdersController()

// 'this' context is lost when Express calls the method
ordersRouter.post('/', controller.createOrder)           // broken
ordersRouter.post('/', controller.createOrder.bind(controller)) // ugly

// Functions just work
export async function createOrder(req, res, next) { ... }
ordersRouter.post('/', createOrder)                      // direct

Services — classes are the stronger choice

// Class-based service: dependencies injected, swappable in tests
export class OrdersService {
  constructor(
    private readonly db: PrismaClient,
    private readonly email: EmailService,
  ) {}
  async createOrder(userId, input) {
    // this.db and this.email — inject mocks in tests
  }
}

// Test — no vi.mock() gymnastics needed
const mockDb    = { order: { create: vi.fn() } }
const mockEmail = { sendOrderConfirmation: vi.fn() }
const service   = new OrdersService(mockDb, mockEmail)

Connection & tool selection

The singleton pattern

// lib/db.ts
import { PrismaClient } from '@prisma/client'
import { config } from '../config/index.js'

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined }

export const db = globalForPrisma.prisma ?? new PrismaClient({
  log: config.NODE_ENV === 'development'
    ? ['query', 'warn', 'error']
    : ['warn', 'error'],
})

if (config.NODE_ENV !== 'production') globalForPrisma.prisma = db

Tool selection

Prisma
Schema-first (schema.prisma), generated client, excellent TypeScript types. Best default for most apps.
Drizzle
Code-first, schema in TypeScript, no codegen step. Queries look like SQL. Growing fast.
Knex
Query builder, not ORM. Good for fine-grained query control or migrating legacy codebases.
Raw pg
Maximum control. Use alongside an ORM for specific performance-critical hot paths.

Repository pattern

The repository wraps your ORM and presents a domain-shaped API to the service. The service never imports Prisma directly — it talks to the repository. If you later swap ORMs, only the repository file changes.

// orders.repository.ts
import { db } from '../../lib/db.js'
import type { PrismaClient, Prisma } from '@prisma/client'

type Tx = Prisma.TransactionClient

export async function findById(id: string, tx: Tx | PrismaClient = db) {
  return (tx).order.findUnique({
    where: { id }, include: { items: true, user: { select: { id: true, email: true } } }
  })
}

export async function findByUser(userId: string, cursor?: string, pageSize = 20) {
  return db.order.findMany({
    where:   { userId },
    take:    pageSize,
    skip:    cursor ? 1 : 0,
    cursor:  cursor ? { id: cursor } : undefined,
    orderBy: { createdAt: 'desc' },
    include: { items: true }
  })
}

export async function createWithItems(userId, input, total, tx: Tx | PrismaClient = db) {
  return (tx).order.create({
    data: {
      userId, total, status: 'PENDING',
      items: { create: input.items.map(i => ({
        productId: i.productId, quantity: i.quantity, unitPrice: i.unitPrice
      }))}
    },
    include: { items: true }
  })
}

export async function updateStatus(id: string, status: string, tx: Tx | PrismaClient = db) {
  return (tx).order.update({ where: { id }, data: { status } })
}
The repository is the one file in a feature that imports from lib/db.ts. This is the file you mock in unit tests — not Prisma directly.

Transactions

Transactions are a service-layer concern. The service knows which operations must succeed or fail atomically. The optional tx parameter pattern bridges them: repository functions work standalone or inside a transaction without any change to their call signature.

// orders.service.ts — the service owns the transaction boundary
export async function createOrder(userId: string, input: CreateOrderInput) {
  for (const item of input.items) {
    const product = await productsRepo.findById(item.productId)
    if (!product)                      throw new NotFoundError(`Product ${item.productId} not found`)
    if (product.stock < item.quantity) throw new ConflictError(`Insufficient stock for ${product.name}`)
  }
  const total = input.items.reduce((s, i) => s + i.quantity * i.unitPrice, 0)

  return db.$transaction(async (tx) => {
    const order = await ordersRepo.createWithItems(userId, input, total, tx)
    for (const item of input.items)
      await productsRepo.decrementStock(item.productId, item.quantity, tx)
    return order
  })
}
The optional tx parameter with a default of the global db client is the key technique. Repository functions become composable into transactions without knowing they're being composed.

Query patterns

Select only what you need

// BAD — fetches password hash, internal flags, timestamps, everything
const user = await db.user.findUnique({ where: { id } })

// GOOD — explicit projection, safe to serialize directly to client
const user = await db.user.findUnique({
  where:  { id },
  select: { id: true, email: true, name: true, createdAt: true }
})

Cursor pagination over offset for large tables

// Offset: SELECT * OFFSET 10000 LIMIT 20 — scans 10,020 rows
export async function findByUserOffset(userId, page, pageSize) {
  return db.order.findMany({
    where: { userId }, skip: (page - 1) * pageSize, take: pageSize,
  })
}

// Cursor: consistent O(1) performance at any depth
export async function findByUserCursor(userId, cursor?, pageSize = 20) {
  return db.order.findMany({
    where:   { userId },
    take:    pageSize,
    skip:    cursor ? 1 : 0,
    cursor:  cursor ? { id: cursor } : undefined,
    orderBy: { createdAt: 'desc' },
  })
}

Avoiding N+1 queries

// BAD — N+1: 1 query for orders, then 1 per order for its items
const orders = await db.order.findMany({ where: { userId } })
for (const order of orders) {
  order.items = await db.orderItem.findMany({ where: { orderId: order.id } })
}

// GOOD — 2 queries total via eager loading
const orders = await db.order.findMany({
  where:   { userId },
  include: { items: true }
})

Migrations

Never mutate your database schema by hand. Migrations are the only mechanism that gives you a reproducible, version-controlled history of schema changes.

Prisma workflow

# Development: edit schema.prisma, then author a migration
npx prisma migrate dev --name add_order_status_index

# CI / Production: apply all pending migrations
npx prisma migrate deploy

Auth strategy overview

The right auth strategy depends on who is making the request. Human users in browsers have different needs from machine clients calling an API.

TokenLifetimeRationale
Access token15 minutesShort enough that a leaked token expires quickly
Refresh token7–30 daysLong enough not to annoy users; stored in DB so it can be revoked
OAuth state param10 minutesOnly needs to survive the redirect roundtrip
Password reset token1 hourShort window reduces exposure if email is compromised
Email verification24 hoursReasonable for user workflow

JWT access + refresh token flow

Two tokens, two jobs. The access token is short-lived and stateless — verify its signature and trust it. The refresh token is long-lived and stored in the database — it can be revoked, and it enables token rotation which detects stolen tokens.

Token reuse detection — critical

On refresh, the token is atomically deleted from the DB before issuing a new one. If the same token is presented twice, it was already deleted — meaning it was either stolen or the client is retrying. Either way, revoke all sessions for that user immediately.

Never store tokens in localStorage. Any XSS vulnerability — including in a third-party script — can silently exfiltrate localStorage contents. The attacker gets a persistent token they can use from anywhere.

Token utilities

Use jose over jsonwebtoken for new projects. It's actively maintained, supports the Web Crypto API, and has a cleaner modern API.

// lib/tokens.ts
import { SignJWT, jwtVerify, type JWTPayload } from 'jose'
import { config } from '../config/index.js'

const accessSecret  = new TextEncoder().encode(config.JWT_ACCESS_SECRET)
const refreshSecret = new TextEncoder().encode(config.JWT_REFRESH_SECRET)

export interface TokenPayload extends JWTPayload {
  userId: string
  role:   string
}

export async function signAccessToken(payload: Omit) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('15m')
    .sign(accessSecret)
}

export async function signRefreshToken(payload: Omit) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('7d')
    .sign(refreshSecret)
}

export async function verifyAccessToken(token: string): Promise {
  const { payload } = await jwtVerify(token, accessSecret)
  return payload as TokenPayload
}

export async function verifyRefreshToken(token: string): Promise {
  const { payload } = await jwtVerify(token, refreshSecret)
  return payload as TokenPayload
}

Login — with timing-attack prevention

export async function login(email: string, password: string) {
  const user = await usersRepo.findByEmail(email)

  // CRITICAL: always run verifyPassword regardless of whether the user exists.
  // Without this, response time differences reveal valid email addresses.
  const hash  = user?.password ?? '$argon2id$v=19$m=65536,t=3,p=4$placeholder'
  const valid = await verifyPassword(hash, password)

  if (!user || !valid) throw new UnauthorizedError('Invalid credentials')
  if (!user.isActive)  throw new UnauthorizedError('Account disabled')

  const payload = { userId: user.id, role: user.role }
  const [accessToken, refreshToken] = await Promise.all([
    signAccessToken(payload), signRefreshToken(payload),
  ])
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  await authRepo.storeRefreshToken(user.id, refreshToken, expiresAt)
  return { accessToken, refreshToken, user: { id: user.id, email: user.email, role: user.role } }
}

Refresh — with rotation and reuse detection

export async function refresh(incomingToken: string) {
  let payload
  try { payload = await verifyRefreshToken(incomingToken) }
  catch { throw new UnauthorizedError('Invalid refresh token') }

  const stored = await authRepo.findAndDeleteRefreshToken(incomingToken)
  if (!stored || stored.expiresAt < new Date()) {
    if (payload?.userId) await authRepo.deleteAllUserTokens(payload.userId)
    throw new UnauthorizedError('Refresh token reused or expired')
  }

  const tokenPayload = { userId: payload.userId, role: payload.role }
  const [accessToken, refreshToken] = await Promise.all([
    signAccessToken(tokenPayload), signRefreshToken(tokenPayload),
  ])
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  await authRepo.storeRefreshToken(payload.userId, refreshToken, expiresAt)
  return { accessToken, refreshToken }
}

    next()
  }
}

API keys

API keys are for machine-to-machine clients — other services, CLI tools, third-party integrations. Hash before storing, return the raw key once and never again.

// lib/apiKeys.ts
import { randomBytes, createHash } from 'crypto'

export function generateApiKey() {
  const raw    = `sk_${randomBytes(32).toString('base64url')}`
  const hash   = createHash('sha256').update(raw).digest('hex')
  const prefix = raw.slice(0, 10)
  return { raw, hash, prefix }
}

export const hashApiKey = (raw: string) =>
  createHash('sha256').update(raw).digest('hex')

API key middleware

export async function authenticateApiKey(req, res, next) {
  const key = req.headers['x-api-key'] as string
  if (!key) return next(new UnauthorizedError())

  const apiKey = await db.apiKey.findUnique({
    where: { hash: hashApiKey(key) }, include: { user: true }
  })
  if (!apiKey?.isActive) return next(new UnauthorizedError())
  if (apiKey.expiresAt && apiKey.expiresAt < new Date())
    return next(new UnauthorizedError('API key expired'))

  db.apiKey.update({ where: { id: apiKey.id },
    data: { lastUsedAt: new Date() } }).catch(() => {})

  req.user = { userId: apiKey.userId, role: apiKey.user.role }
  next()
}
Store prefix (first 10 chars) alongside the hash. This lets you show users which key is which in your UI ("sk_a3f9b2...") without storing anything that could reconstruct the key.

Security defence layers

No single control stops all attacks. Security is a set of overlapping layers — if one fails, others catch what slipped through.

An attacker trying to reach your database has to pass through all five layers. Input validation stops injection. Auth middleware stops unauthorized access. Parameterized queries stop SQL injection even if validation somehow fails.

Content Security Policy (CSP)

CSP is your primary browser-side defence against XSS. It tells the browser which sources are allowed to load scripts, styles, and other resources. Anything not explicitly allowed is blocked.

Never use 'unsafe-inline' on scriptSrc or 'unsafe-eval' in production. It completely neutralizes CSP's XSS protection.

Injection attacks

Injection is the #1 category in the OWASP Top 10. SQL, NoSQL, command, and LDAP injection all share the same root cause: unsanitized user input interpreted as code. The fix is always the same: never concatenate user input into commands or queries.

SQL injection

// VULNERABLE — attacker sends: email = "' OR 1=1; DROP TABLE users; --"
const user = await db.query(`SELECT * FROM users WHERE email = '${req.body.email}'`)

// SAFE — parameterized query
const user = await db.query('SELECT * FROM users WHERE email = $1', [req.body.email])

// Prisma template literal IS safe — parameterizes automatically
const user = await db.$queryRaw`SELECT * FROM users WHERE email = ${req.body.email}`

// UNSAFE even with Prisma — direct interpolation, no parameterization
const user = await db.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${req.body.email}'`)

NoSQL injection (MongoDB)

// VULNERABLE — attacker sends { "email": { "$gt": "" } }
await db.collection('users').findOne({ email: req.body.email })

// SAFE — Zod validates that email is a string before it reaches the DB
const { email } = z.object({ email: z.string().email() }).parse(req.body)
await db.collection('users').findOne({ email })

Command injection

// VULNERABLE
exec(`convert ${req.body.filename} output.png`)

// SAFE — execFile with argument array: no shell, no injection
execFile('convert', [sanitizedFilename, 'output.png'])

Cross-Site Scripting (XSS)

XSS lets attackers inject scripts into pages viewed by other users. A successful attack can steal tokens, impersonate users, and exfiltrate data invisibly.

Layer 1 — Zod input validation

Reject malformed input at the controller boundary. A name field expecting text should reject HTML via an allowlist regex.

Layer 2 — output escaping

Template engines (Handlebars, EJS, Nunjucks) auto-escape output by default. Treat any unescaped output tag as a deliberate decision requiring code review justification.

Layer 3 — CSP with nonces

Even if malicious content is stored and rendered, a correctly configured CSP with nonces prevents it from executing.

Layer 4 — httpOnly cookies

Refresh tokens in httpOnly cookies are inaccessible to JavaScript entirely. A successful XSS attack can't steal the refresh token.

DOM sinks — never pass user data to these

element.innerHTML   = userInput   // DANGEROUS — parses and executes HTML/JS
eval(userInput)                   // DANGEROUS
new Function(userInput)           // DANGEROUS

element.textContent = userInput   // SAFE — treated as text, never executed

How supply chain attacks work

A supply chain attack doesn't target your code — it targets the code your code depends on. The attacker poisons a package upstream and it gets pulled in automatically.

1
Target selection

Attackers identify high-value maintainers — people who own packages with hundreds of millions of weekly downloads.

2
Phishing for credentials

Attackers send email from lookalike domains. The fake page captures both the login password and the live TOTP code in real time, using them immediately before they expire.

3
Malicious publish

With stolen credentials, attackers publish new patch versions containing a postinstall script that exfiltrates environment variables, SSH keys, and installs a persistent backdoor.

4
Self-replicating spread

A worm uses npm tokens stolen from infected machines to compromise other maintainers' accounts and publish malicious versions of their packages.

5
Blast radius

Any developer or CI runner that ran npm install (not npm ci) during the window pulled the compromised version.

The difference between npm install and npm ci was the difference between being compromised and not. npm ci installs exactly what's in package-lock.json — period.

Why TOTP isn't enough

Hardware security keys (WebAuthn/FIDO2) cannot be phished this way. The key signs a challenge bound to the legitimate domain — a fake login page cannot extract a usable credential. A TOTP code, by contrast, can be relayed in real time.

CORS mental model

CORS is a browser enforcement mechanism — not a server security feature in the traditional sense. When you configure CORS on your Express server, you are telling browsers which cross-origin JavaScript is allowed to read your API responses. You are not blocking requests from reaching your server.

A direct consequence: CORS does nothing to protect against requests made by non-browser clients. curl, Postman, other servers, and mobile apps all ignore CORS headers entirely. Your real server-side protection is authentication and authorization.

What "origin" means precisely

https://app.example.com       ← origin A
http://app.example.com        ← DIFFERENT (different protocol)
https://api.example.com       ← DIFFERENT (different subdomain)
https://app.example.com:8080  ← DIFFERENT (different port)
https://app.example.com/path  ← SAME ORIGIN (path is irrelevant)

Simple vs preflighted requests

Preflighted requests — DELETE, PUT, PATCH, or any request with custom headers like Authorization — trigger an OPTIONS request first. The browser asks your server: "will you allow this cross-origin DELETE with these headers?" If it doesn't respond correctly, the real request never fires.

The cors package handles OPTIONS preflight automatically when applied as app-level middleware via app.use(cors(options)). If you apply CORS per-route, you must also add app.options('*', cors(options)) globally.

CORS configuration

// config/cors.ts
import cors from 'cors'
import { config } from './index.js'

const allowedOrigins = new Set(
  config.ALLOWED_ORIGINS.split(',').map(o => o.trim())
)

export const corsOptions: cors.CorsOptions = {
  origin(origin, callback) {
    if (!origin) return callback(null, true)
    if (allowedOrigins.has(origin)) callback(null, true)
    else callback(new Error(`Origin ${origin} not allowed by CORS policy`))
  },
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-Request-ID', 'X-CSRF-Token'],
  credentials: true,
  maxAge: 86400,
}

app.use(cors(corsOptions))
app.options('*', cors(corsOptions))
Never combine origin: '*' with credentials: true. Browsers block this because it would mean any website can make authenticated requests on behalf of your logged-in users.

Separate policy for public endpoints

const publicCors  = cors({ origin: '*', methods: ['GET'] })
const privateCors = cors(corsOptions)

app.use('/api/v1/public', publicCors)
app.use('/api/v1',        privateCors)

CORS patterns & pitfalls

Vary: Origin — critical for CDN caching

When you serve different CORS headers for different origins, you must include Vary: Origin in responses. Without it, a CDN may cache a response for origin A and serve it with origin A's CORS headers to origin B. The cors package adds this automatically when you use a function for the origin option.

Subdomain allowlisting — avoid regex

// DANGEROUS — this regex matches evil.example.com.attacker.com
origin: /example\.com$/

// SAFE — explicit Set of only the known good origins
const allowed = new Set(['https://app.example.com', 'https://admin.example.com'])

Testing your CORS policy with curl

# Test a preflight for DELETE with Authorization header
curl -I -X OPTIONS \
  -H 'Origin: https://app.example.com' \
  -H 'Access-Control-Request-Method: DELETE' \
  -H 'Access-Control-Request-Headers: Authorization' \

Why Nginx in front of Node

Node.js is excellent at running application logic but is not designed to be a front-line internet server. Nginx is purpose-built for that role.

TLS termination
Nginx handles HTTPS, certificate management, and cipher negotiation. Node sees only plain HTTP internally.
Static file serving
Nginx serves static files directly from disk at kernel speed, bypassing Node entirely.
Connection management
Nginx handles thousands of slow or idle connections. It only opens a connection to Node when a request is ready.
Rate limiting
Nginx drops malicious traffic before it reaches your app, independent of Node's load.
Load balancing
Distributes traffic across multiple Node processes to use all CPU cores.
Information hiding
Strips server fingerprinting headers.

        expires 1y;
        add_header Cache-Control "public, immutable";
        try_files $uri =404;

    location /api/ {
        proxy_pass         http://nodejs_app;
        proxy_http_version 1.1;
        proxy_set_header   Connection        "";
        proxy_set_header   Host              $host;
        proxy_set_header   X-Real-IP         $remote_addr;
        proxy_set_header   X-Forwarded-For   $proxy_add_x_forwarded_for;

What is observability

Observability is the ability to understand the internal state of a system by examining its external outputs — without needing to add new instrumentation every time a new question arises. It's distinct from monitoring, which tells you when something is broken. Observability lets you understand why.

The practical difference: monitoring says "error rate is 5%." Observability says "error rate is 5%, all from user accounts created before the migration, all hitting the /orders endpoint, all with a specific product ID in the payload, all originating from mobile clients." You can answer that second question without deploying new code because the data is already there.

Why it matters more as your app grows

A single Express app on one server is debuggable with logs and a process monitor. Once you have multiple services, multiple instances, background workers, and queues, a single failed request may have touched a dozen components. Without distributed traces you are guessing at causality.

Even for a single-service app, observability pays off during incidents. The difference between a 15-minute outage and a 3-hour one is often whether the person on call can answer "which requests are failing and why" in 30 seconds or 30 minutes.

The three pillars

Observability data comes in three forms. Each answers different questions. You need all three — they complement rather than replace each other.

PillarQuestion it answersTool for storageRetention
TracesWhy did this request take 4 seconds?Jaeger, Tempo, HoneycombDays to weeks
MetricsIs p99 latency trending upward?Prometheus, VictoriaMetricsWeeks to months
LogsWhat was the exact error message at 14:32?Loki, Elasticsearch, CloudWatchDays to months

How they work together in practice

An alert fires: p99 latency for POST /orders exceeds 2 seconds. That's a metric alert. You click into the metric and find it started 20 minutes ago. You open the distributed trace for one of the slow requests — the trace shows 1.8 seconds spent inside ordersRepo.createWithItems. You click the span and see it has a db.statement attribute — a full table scan on a missing index. You jump to the logs correlated to that trace ID and see a Prisma warning logged three days ago that you missed. Root cause found in under 5 minutes.

OpenTelemetry overview

OpenTelemetry (OTel) is the CNCF standard for producing, collecting, and exporting telemetry data. It is vendor-neutral — you instrument your app once and can send data to any compatible backend (Jaeger, Datadog, Honeycomb, Grafana, etc.) by changing configuration, not code.

Key concepts

Span
A single unit of work — one DB query, one HTTP call, one function. Has a start time, end time, attributes, and status.
Trace
A collection of spans linked by a trace ID. Represents the full lifecycle of one request, potentially across many services.
Context propagation
The trace ID travels with the request via HTTP headers (traceparent). Downstream services add their spans to the same trace.
Exporter
The component that sends telemetry data to a backend. OTLP (OpenTelemetry Protocol) is the universal format — supported by all major backends.
Resource
Attributes that describe the entity producing telemetry — service name, version, environment. Attached to all spans and metrics from a process.
Sampler
Decides which traces to record. Sampling 100% of traffic in production is expensive — a rate-based or tail-based sampler reduces volume while keeping interesting traces.
OpenTelemetry is now stable for traces and metrics. The logging signal is GA as of late 2024. Use it for all three pillars rather than mixing OTel traces with a separate, unconnected logging library — correlation comes for free when they're from the same SDK.

instrumentation.ts

This file configures the OTel SDK and must be loaded before everything else via --import. It defines three things: your service's identity (resource), where to send data (exporters), and what to instrument (instrumentations).

// src/instrumentation.ts
import { NodeSDK } from '@opentelemetry/sdk-node'
import { Resource } from '@opentelemetry/resources'
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION } from '@opentelemetry/semantic-conventions'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http'
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics'
import { BatchSpanProcessor, ParentBasedSampler, TraceIdRatioBasedSampler } from '@opentelemetry/sdk-trace-base'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'

const isProd = process.env.NODE_ENV === 'production'

const sdk = new NodeSDK({
  // Service identity — appears in all traces and metrics
  resource: new Resource({
    [ATTR_SERVICE_NAME]:    process.env.SERVICE_NAME    ?? 'my-api',
    [ATTR_SERVICE_VERSION]: process.env.SERVICE_VERSION ?? '0.0.0',
    'deployment.environment': process.env.NODE_ENV ?? 'development',
  }),

  // Trace exporter — sends to OTel Collector or directly to backend
  spanProcessor: new BatchSpanProcessor(
    new OTLPTraceExporter({
      url: process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
        ?? 'http://localhost:4318/v1/traces',
    }),
    {
      maxQueueSize:        2048,
      maxExportBatchSize:  512,
      scheduledDelayMillis: 5000,
    }
  ),

  // Metrics exporter — pushes on a schedule
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: process.env.OTEL_EXPORTER_OTLP_METRICS_ENDPOINT
        ?? 'http://localhost:4318/v1/metrics',
    }),
    exportIntervalMillis: 30_000,  // push every 30s
  }),

  // Sampling — 100% in dev, 10% in prod (adjust per traffic volume)
  sampler: isProd
    ? new ParentBasedSampler({
        root: new TraceIdRatioBasedSampler(
          parseFloat(process.env.OTEL_SAMPLE_RATE ?? '0.1')
        )
      })
    : undefined,  // undefined = sample everything (default)

  // Auto-instrument Express, HTTP, pg, Prisma, Redis, etc.
  instrumentations: [
    getNodeAutoInstrumentations({
      // Disable noisy fs instrumentation
      '@opentelemetry/instrumentation-fs': { enabled: false },
      // Ignore health check spans to reduce noise
      '@opentelemetry/instrumentation-http': {
        ignoreIncomingRequestHook: (req) =>
          req.url === '/health',
      },
    }),
  ],
})

process.on('SIGINT',  () => sdk.shutdown())

Auto-instrumentation

With getNodeAutoInstrumentations() loaded via --import, you get traces for free across the entire request lifecycle — no code changes to your app required.

What gets instrumented automatically

HTTP layer
Every incoming request becomes a root span. Method, route, status code, response time all captured as attributes.
Express routing
Each middleware and route handler gets a child span. You can see exactly which middleware is slow.
Prisma / pg
Every database query gets a span with the SQL statement, table name, and duration. N+1 queries are immediately visible.
Outbound HTTP
Calls to external APIs (fetch, axios, got) get spans with target URL, method, and status. Propagates trace context downstream.
Redis
Cache hits and misses, command names, and latency — all visible as child spans.
DNS / TCP
Connection establishment time captured — useful for diagnosing cold-start latency and connection pool exhaustion.

Custom spans

Auto-instrumentation covers infrastructure boundaries (HTTP, DB, cache). Custom spans cover what happens inside your application logic — business operations, validation steps, external integrations that aren't auto-instrumented.

Basic custom span

// features/orders/orders.service.ts
import { SpanStatusCode } from '@opentelemetry/api'
import { getTracer } from '../../lib/tracer.js'

export async function createOrder(userId: string, input: CreateOrderInput) {
  const tracer = getTracer()

  return tracer.startActiveSpan('orders.create', async (span) => {
    try {
      // Add attributes — searchable dimensions in your tracing backend
      span.setAttributes({
        'order.user_id':    userId,
        'order.item_count': input.items.length,
        'order.total_value': input.items.reduce(
          (s, i) => s + i.quantity * i.unitPrice, 0
        ),
      })

      // Child spans for sub-operations
      const validationResult = await tracer.startActiveSpan(
        'orders.validate_stock',
        async (childSpan) => {
          try {
            const result = await validateStock(input.items)
            childSpan.setStatus({ code: SpanStatusCode.OK })
            return result
          } catch (err) {
            childSpan.recordException(err as Error)
            childSpan.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message })
            throw err
          } finally {
            childSpan.end()
          }
        }
      )

      const order = await createOrderInDB(userId, input)

      // Add outcome attributes after the operation
      span.setAttributes({ 'order.id': order.id, 'order.status': order.status })
      span.setStatus({ code: SpanStatusCode.OK })
      return order
    } catch (err) {
      span.recordException(err as Error)
      span.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message })
      throw err
    } finally {
      span.end()  // always end the span — missing end() = missing trace
    }
  })
}
Always call span.end() — put it in a finally block. A span that is never ended is either dropped or held in memory indefinitely depending on the SDK configuration.

Span attribute conventions

// Use namespaced dot-notation attributes

Context propagation

Context propagation is what turns a collection of spans into a distributed trace. When your service calls another service, it injects the current trace ID into the outgoing request headers. The downstream service reads those headers and creates its spans as children of the same trace.

How it works automatically

# OTel auto-instrumentation handles W3C Trace Context headers automatically
# Outgoing fetch/http.request calls get:
# traceparent: 00-{traceId}-{spanId}-01

# The downstream service reads this header and links its spans:
# GET /inventory (downstream)
#   └── parent: your POST /orders span ID

Manual context injection — for queues, workers, cron jobs

Auto-instrumentation only covers HTTP. If you publish a job to a queue, the trace context needs to be carried in the job payload so the worker can continue the trace.

import { context, propagation, trace } from '@opentelemetry/api'

// PRODUCER — inject trace context into the job payload
export async function queueOrderNotification(orderId: string) {
  const carrier: Record = {}
  propagation.inject(context.active(), carrier)
  // carrier now contains: { traceparent: '00-...', tracestate: '...' }

  await queue.publish('order.notifications', {
    orderId,
    _otelContext: carrier,   // carry context in the payload
  })
}

// CONSUMER (worker process) — restore context from payload
export async function handleOrderNotification(job: Job) {
  const parentContext = propagation.extract(
    context.active(),
    job.data._otelContext ?? {}
  )

  // Run within the restored context — spans created here are children
  // of the original POST /orders trace
  await context.with(parentContext, async () => {
    const tracer = getTracer()
    return tracer.startActiveSpan('notifications.send', async (span) => {
      try {
        await sendNotification(job.data.orderId)
        span.setStatus({ code: SpanStatusCode.OK })
      } finally {
        span.end()
      }
    })
  })
}
Without context propagation in queues and workers, every background job appears as an isolated trace with no connection to the request that triggered it. With it, you see the full picture: request → queue publish → worker execution → DB write, all in one trace.

Why histograms matter more than averages

An average latency of 100ms can hide the fact that 1% of requests take 10 seconds. Histograms let you query percentiles — p50, p95, p99. The difference between p50 and p99 latency tells you whether you have a consistent problem or a tail latency problem, which leads to completely different fixes.

Recording measurements in the service

// features/orders/orders.service.ts

  const order = await createOrderInDB(userId, input)

  // Record metrics with attributes (labels in Prometheus terms)

    'order.region':   input.region,

  })

Attribute cardinality is critical. A unique attribute value per user ID would create millions of metric series and crash your Prometheus. Attributes should have low cardinality — status codes, regions, plan types, boolean flags. Never use user IDs, order IDs, or request IDs as metric attributes.

Structured logging

We covered pino setup in the Logging section — this tab covers how logging integrates with the OTel observability stack specifically. The critical addition is injecting trace context into every log line.

Why trace context in logs matters

When an error occurs, you'll find it in your log aggregator. Without trace context you're stuck filtering by time and endpoint hoping to find related entries. With a traceId on every log line, you click the trace ID and jump directly to the full distributed trace in Jaeger, seeing exactly what the code was doing across every service at the moment the error was logged.

Injecting trace context into pino logs

// middleware/requestLogger.ts
import { trace, context } from '@opentelemetry/api'

export function requestLogger(req, res, next) {
  const span     = trace.getActiveSpan()
  const spanCtx  = span?.spanContext()

  // Inject trace context — these fields appear in every log line
  // for this request, enabling jump-to-trace from your log aggregator
  req.log = logger.child({
    traceId: spanCtx?.traceId,
    spanId:  spanCtx?.spanId,
    requestId: req.id,
  })

  const start = Date.now()
  res.on('finish', () => {
    req.log.info({
      method: req.method,
      path:   req.path,
      status: res.statusCode,
      ms:     Date.now() - start,
    })
  })
  next()
}

Sampling strategy

At scale, recording 100% of traces is expensive and usually unnecessary — most requests succeed in normal time and aren't interesting to store. Sampling reduces volume while ensuring you keep the traces that matter.

Head sampling — decide at trace start

// instrumentation.ts — sample 10% of all traces
// Decision made at the first span before any child spans are created
sampler: new ParentBasedSampler({
  root: new TraceIdRatioBasedSampler(0.1)  // 10%
})

// ParentBasedSampler: if an upstream service sampled the trace,
// this service keeps it too — prevents incomplete traces

Head sampling is simple and cheap, but blind — you might drop the 10% that contains your errors. Use it only for very high-volume services where you need cost control above all else.

Tail sampling — decide after the trace is complete

Tail sampling (in the Collector) waits for the full trace to arrive, then applies intelligent rules. This is almost always the right approach in production.

# In the OTel Collector config (see Collector tab):
# Keep 100% of error traces
# Keep 100% of traces over 1000ms
# Keep 10% of everything else
tail_sampling:
  decision_wait: 10s    # wait up to 10s for all spans to arrive
  policies:
    - name: keep-errors
      type: status_code
      status_code: { status_codes: [ERROR] }
    - name: keep-slow
      type: latency
      latency: { threshold_ms: 1000 }
    - name: sample-normal
      type: probabilistic
      probabilistic: { sampling_percentage: 10 }

Always sample certain routes

// In instrumentation.ts — override sampling for specific routes
// Useful for payment flows, auth flows where you always want full traces
import { AlwaysOnSampler } from '@opentelemetry/sdk-trace-base'

const customSampler = {
  shouldSample(ctx, traceId, spanName, kind, attrs) {
    // Always sample payment and auth routes
    const url = attrs?.['http.url'] as string ?? ''
    if (url.includes('/payment') || url.includes('/auth')) {
      return { decision: SamplingDecision.RECORD_AND_SAMPLED }
    }
    // Default ratio sampling for everything else
    return ratioSampler.shouldSample(ctx, traceId, spanName, kind, attrs)
  },
  toString: () => 'CustomSampler'
}

Alerting & SLOs

Alerts should wake someone up only when a user-visible problem exists. Alert on symptoms, not causes — alert on error rate, not CPU usage. CPU spiking might be fine; users getting errors is never fine.

Service Level Objectives (SLOs)

An SLO is a target for how reliable your service should be. Alerting fires when you're at risk of breaching it. This is a more disciplined approach than ad-hoc threshold alerts.

# Example SLOs for an API service:
# - 99.9% of requests succeed (error rate < 0.1%)
# - 95% of requests complete in under 200ms
# - 99% of requests complete in under 1000ms

# Prometheus recording rules (pre-compute for efficiency)
groups:
  - name: slo_rules
    interval: 30s
    rules:
      # Request success rate over 5 minutes
      - record: job:request_success_rate:5m
        expr: |
          1 - (
            rate(http_server_duration_count{http_status_code=~"5.."}[5m])
            /
            rate(http_server_duration_count[5m])
          )

Prometheus alerting rules

groups:
  - name: api_alerts
    rules:
      # Error rate above 1% for 5 minutes — page immediately
      - alert: HighErrorRate
        expr: job:request_success_rate:5m < 0.99
        for: 5m
        labels:   { severity: critical }
        annotations:
          summary: "Error rate {{ $value | humanizePercentage }} on {{ $labels.job }}"

      # p99 latency above 2 seconds for 10 minutes
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            rate(http_server_duration_bucket[5m])
          ) > 2
        for: 10m
        labels:   { severity: warning }
        annotations:
          summary: "p99 latency {{ $value }}s on {{ $labels.http_route }}"

      # No requests for 5 minutes during business hours (possible outage)
      - alert: NoTraffic
        expr: rate(http_server_duration_count[5m]) == 0
        for: 5m
        labels:   { severity: critical }

Observability checklist

Instrumentation

OTel SDK loaded via --import before all other modules

instrumentation.ts loaded first. sdk.start() called synchronously, not inside async.

Service resource attributes set: name, version, environment

All telemetry from this process is identifiable. Essential when running multiple services.

sdk.shutdown() called on SIGTERM and SIGINT

Ensures in-flight telemetry is flushed before process exits. Without this you lose the last batch of spans.

Health check route excluded from trace instrumentation

Eliminates noise — health checks are high-frequency, always successful, and never interesting.

Custom spans on all significant business operations

Auto-instrumentation covers infrastructure. Custom spans cover domain — order creation, payment processing, email dispatch.

span.end() always in a finally block

A span that never ends is silently dropped or leaks memory. No exceptions.

Metrics

RED metrics visible per route: rate, error rate, p99 latency

Minimum viable dashboard for any API service. First stop in every incident.

Metric attributes are low cardinality

No user IDs, order IDs, or UUIDs as attributes. Status codes, regions, plan types only.

DB connection pool depth tracked as ObservableGauge

Pool exhaustion is a common production failure mode — you want to see it coming.

Logs

traceId injected into every log line

Enables log-to-trace correlation. Without this, logs and traces are separate islands.

Security events logged with structured fields

auth.login.failure, auth.token.reuse, authz.forbidden — all with ip, userId, requestId.

No PII or secrets in logs

Passwords, tokens, card numbers, SSNs never appear in log output under any conditions.

Alerting

Alerts fire on symptoms, not causes

Error rate and latency, not CPU or memory. Alert on what users experience.

Every critical alert has a runbook

A link in the alert annotation to a document explaining: what this means, how to investigate, known causes, how to fix.

Tail sampling in production — always keep errors and slow traces

Head sampling at a ratio drops errors randomly. Tail sampling in the Collector ensures you always keep the interesting traces.
